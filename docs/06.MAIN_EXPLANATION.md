# Understanding `main.py` — The Reception Desk of Your RAG System

## The Big Picture: What Problem Does This Solve?

You've built a powerful RAG pipeline (`rag_pipeline.py`) that can ingest documents and answer questions. But right now, the only way to use it is by writing Python code:

```python
pipeline = RAGPipeline()
pipeline.ingest("data/sample_texts/")
result = pipeline.query("What is machine learning?")
```

That works for developers, but it's not how people normally interact with software. People want to type a command in their terminal and get a result. That's what `main.py` does — it's the **command-line interface (CLI)** that translates terminal commands into pipeline actions.

> **Analogy: The Reception Desk**
>
> You've built a whole office (the RAG pipeline) with specialists:
> - A mail room (document loader)
> - A prep station (chunker)
> - A translator (embedder)
> - A filing cabinet (vector store)
> - A research assistant (LLM)
>
> But there's no reception desk — no one to greet visitors and direct them. A visitor walks in and says "I want to store some documents" or "I have a question." The receptionist (main.py) listens, understands what they want, and directs them to the right department.
>
> Without the reception desk, visitors would need to walk through the building themselves and know exactly which doors to open. With it, they just say what they need and the receptionist handles the rest.

---

## Where Does It Fit in the RAG Pipeline?

```
USER (terminal)
    │
    ▼
  main.py    ← YOU ARE HERE (the CLI entry point)
    │
    ▼
  rag_pipeline.py  (orchestrates everything)
    │
    ├── document_loader.py
    ├── chunker.py
    ├── embedder.py
    └── vector_store.py
```

`main.py` sits **above** the pipeline. It's the outermost layer — the part that touches the user directly. It translates human-typed terminal commands into Python function calls on the `RAGPipeline`.

```
User types:    uv run rag-basics ingest --data-dir data/sample_texts/
                                    │
main.py:       Parses this into:    args.command = "ingest"
                                    args.data_dir = "data/sample_texts/"
                                    │
rag_pipeline:  Calls:               pipeline.ingest("data/sample_texts/")
```

> - A **key cabinet** (`load_dotenv`) — to unlock API keys stored in a safe (`.env` file)
> - A **direct line to the office manager** (`RAGPipeline`) — to pass along instructions
> - A **filing room key** (`ChromaVectorStore`) — to open the right filing cabinet

### 2. `load_dotenv()` — Loading Secret Keys

```python
def main():
    load_dotenv()
```

This is the **first thing that happens**. `load_dotenv()` reads a file called `.env` in your project root and loads its contents as environment variables.

Your `.env` file looks something like:
```
OPENAI_API_KEY=sk-abc123def456...
```

After `load_dotenv()` runs, Python can access this value via `os.environ["OPENAI_API_KEY"]` — which is exactly what the `OpenAI()` client (used by the embedder and LLM) looks for automatically.

> **Analogy: Opening the Safe**
>
> Before the office can do any work, someone needs to open the safe and take out the keys (API keys). The `.env` file is the safe. `load_dotenv()` opens it. Without this step, the embedder and LLM would fail with "authentication error" because they can't find the API key.

**Why not hardcode the key?**

```python
# NEVER do this:
client = OpenAI(api_key="sk-abc123def456...")
```

Hardcoded keys end up in git history, get shared accidentally, and become a security nightmare. The `.env` file is listed in `.gitignore` so it never gets committed.

### 3. `argparse` — Understanding What the User Typed

```python
parser = argparse.ArgumentParser(description="RAG Basics - Learn RAG by doing")
subparsers = parser.add_subparsers(dest="command", help="Available commands")
```

This is Python's built-in system for parsing command-line arguments. Let's understand it step by step.

> **Analogy: The Intake Form**
>
> When you visit a doctor's office, the receptionist hands you a form:
> - **Name:** ___________
> - **Reason for visit:** ☐ Check-up  ☐ Emergency  ☐ Follow-up
> - **Insurance:** ___________
>
> `argparse` creates a similar "form" for the terminal. It defines what fields exist, which are required, and what the defaults are. When the user runs a command, argparse fills in the form and hands it to the program.

#### What is `ArgumentParser`?

```python
parser = argparse.ArgumentParser(description="RAG Basics - Learn RAG by doing")
```

This creates the top-level parser — the "form" itself. The `description` shows up when a user runs `--help`:

```bash
$ uv run rag-basics --help
usage: rag-basics [-h] {ingest,query} ...

RAG Basics - Learn RAG by doing     ← this is the description

positional arguments:
  {ingest,query}  Available commands
```

#### What are subparsers?

```python
subparsers = parser.add_subparsers(dest="command", help="Available commands")
```

Subparsers let you create **sub-commands** — like how `git` has `git commit`, `git push`, `git pull` as separate commands, each with their own arguments.

Our CLI has two sub-commands:
- `rag-basics ingest` — store documents
- `rag-basics query` — ask questions

`dest="command"` means the chosen sub-command will be stored in `args.command`.

> **Analogy: Different Forms for Different Departments**
>
> The receptionist doesn't use one form for everything. There's a "New Patient" form (ingest) and a "Question for the Doctor" form (query). Each form has different fields. Subparsers let you define separate "forms" for each command.

### 4. The Ingest Command — Defining the "Store Documents" Form

```python
# Ingest command
ingest_parser = subparsers.add_parser("ingest", help="Ingest documents into the vector store")
ingest_parser.add_argument("--data-dir", required=True, help="Directory containing documents")
ingest_parser.add_argument("--pattern", default="*.txt", help="File pattern to match (default: *.txt)")
ingest_parser.add_argument("--chunk-size", type=int, default=500, help="Chunk size in characters (default: 500)")
ingest_parser.add_argument("--overlap", type=int, default=50, help="Chunk overlap in characters (default: 50)")
ingest_parser.add_argument("--collection", default="rag-basics", help="ChromaDB collection name")
ingest_parser.add_argument("--persist-dir", default="./vectordb_data/chromadb", help="ChromaDB persist directory")
```

This defines all the options a user can pass to the `ingest` command.

#### Each argument explained

| Argument | Required? | Default | What It Controls | Example |
|----------|-----------|---------|-----------------|---------|
| `--data-dir` | Yes | — | Where to find the documents | `--data-dir data/sample_texts/` |
| `--pattern` | No | `"*.txt"` | Which files to load | `--pattern "*.pdf"` |
| `--chunk-size` | No | `500` | Characters per chunk | `--chunk-size 1000` |
| `--overlap` | No | `50` | Overlap between chunks | `--overlap 100` |
| `--collection` | No | `"rag-basics"` | Name of the ChromaDB collection | `--collection my-project` |
| `--persist-dir` | No | `"./vectordb_data/chromadb"` | Where to save the database on disk | `--persist-dir /data/db` |

> **Why `required=True` for `--data-dir` but not for others?**
>
> You MUST tell the program where your documents are — there's no sensible default. But chunk size, overlap, collection name, etc. all have reasonable defaults that work for most cases. The user only needs to override them if they want different behavior.

#### How the user runs this command

```bash
# Minimal (just the required argument):
uv run rag-basics ingest --data-dir data/sample_texts/

# With all options:
uv run rag-basics ingest \
    --data-dir data/sample_texts/ \
    --pattern "*.txt" \
    --chunk-size 1000 \
    --overlap 100 \
    --collection my-project \
    --persist-dir ./my_database
```

#### The `--` prefix means optional

In argparse:
- Arguments starting with `--` (like `--data-dir`) are **optional** (unless `required=True`)
- Arguments without `--` (like `question` in the query command) are **positional** — they're required and their order matters

### 5. The Query Command — Defining the "Ask a Question" Form

```python
# Query command
query_parser = subparsers.add_parser("query", help="Ask a question using RAG")
query_parser.add_argument("question", help="The question to ask")
query_parser.add_argument("--top-k", type=int, default=3, help="Number of chunks to retrieve (default: 3)")
query_parser.add_argument("--model", default="gpt-4o-mini", help="OpenAI model for generation")
query_parser.add_argument("--collection", default="rag-basics", help="ChromaDB collection name")
query_parser.add_argument("--persist-dir", default="./vectordb_data/chromadb", help="ChromaDB persist directory")
```

| Argument | Required? | Default | What It Controls | Example |
|----------|-----------|---------|-----------------|---------|
| `question` | Yes (positional) | — | The question to ask | `"What is machine learning?"` |
| `--top-k` | No | `3` | How many chunks to retrieve | `--top-k 5` |
| `--model` | No | `"gpt-4o-mini"` | Which LLM to use | `--model gpt-4o` |
| `--collection` | No | `"rag-basics"` | Which ChromaDB collection to search | `--collection my-project` |
| `--persist-dir` | No | `"./vectordb_data/chromadb"` | Where the database is stored | `--persist-dir /data/db` |

Notice `question` has **no `--` prefix** — it's a **positional argument**. The user just types the question directly:

```bash
# The question is positional — no --question flag needed:
uv run rag-basics query "What is machine learning?"

# With options:
uv run rag-basics query "What is machine learning?" --top-k 5 --model gpt-4o
```

> **Why is `question` positional but `--top-k` is optional?**
>
> A question is the whole purpose of the query command — you can't query without one. But `top_k` and `model` have sensible defaults. Making `question` positional also makes the command more natural to type:
>
> ```bash
> # Natural:
> rag-basics query "What is AI?"
>
> # Clunky:
> rag-basics query --question "What is AI?"
> ```

### 6. Parsing Arguments — Reading the Filled-In Form

```python
args = parser.parse_args()

if args.command is None:
    parser.print_help()
    sys.exit(1)
```

#### `parser.parse_args()`

This is where the magic happens. `parse_args()` reads what the user typed in the terminal and converts it into a Python object:

```bash
uv run rag-basics ingest --data-dir data/sample_texts/ --chunk-size 1000
```

Becomes:

```python
args.command = "ingest"
args.data_dir = "data/sample_texts/"
args.pattern = "*.txt"           # default
args.chunk_size = 1000           # user specified
args.overlap = 50                # default
args.collection = "rag-basics"   # default
args.persist_dir = "./vectordb_data/chromadb"  # default
```

> **Analogy: The Receptionist Reads the Form**
>
> The visitor fills out the form and hands it back. The receptionist reads it: "OK, you want the 'ingest' department, your documents are in 'data/sample_texts/', and you want chunk size 1000. Everything else is standard." The filled-in form is the `args` object.

#### The "no command" guard

```python
if args.command is None:
    parser.print_help()
    sys.exit(1)
```

If the user just types `rag-basics` without `ingest` or `query`, we show the help text and exit. `sys.exit(1)` exits with code 1, which in Unix convention means "something went wrong" (code 0 means success).

> **Analogy:** A visitor walks in and just stares at the receptionist without saying anything. The receptionist points to the menu board (help text) and says: "You need to tell me what you want."

### 7. The Ingest Branch — Executing "Store Documents"

```python
if args.command == "ingest":
    store = ChromaVectorStore(collection_name=args.collection, persist_dir=args.persist_dir)
    pipeline = RAGPipeline(
        vector_store=store,
        chunk_size=args.chunk_size,
        chunk_overlap=args.overlap,
    )
    count = pipeline.ingest(args.data_dir, pattern=args.pattern)
    print(f"Ingested {count} chunks into collection '{args.collection}'")
```

This is where the user's terminal command gets translated into actual pipeline actions:

| Step | Code | What Happens |
|------|------|-------------|
| 1 | `ChromaVectorStore(...)` | Open (or create) the database with the user's settings |
| 2 | `RAGPipeline(...)` | Create a pipeline wired to that database, with user's chunk settings |
| 3 | `pipeline.ingest(...)` | Load → chunk → embed → store all documents |
| 4 | `print(...)` | Tell the user how many chunks were stored |

> **Analogy: The Receptionist Relays to the Office**
>
> 1. "Filing room, open cabinet 'rag-basics'" → `ChromaVectorStore(...)`
> 2. "Office manager, here's the setup" → `RAGPipeline(...)`
> 3. "Go process the documents from this folder" → `pipeline.ingest(...)`
> 4. "Done! We processed 47 chunks." → `print(...)`

#### Why create `ChromaVectorStore` separately?

We create the store outside the pipeline so we can pass the user's custom `collection_name` and `persist_dir`. If we just did `RAGPipeline()` with defaults, the user couldn't customize where the database lives.

---

### 8. The Query Branch — Executing "Ask a Question"

```python
elif args.command == "query":
    store = ChromaVectorStore(collection_name=args.collection, persist_dir=args.persist_dir)
    pipeline = RAGPipeline(
        vector_store=store,
        llm_model=args.model,
    )
    result = pipeline.query(args.question, top_k=args.top_k)

    print(f"\nQuestion: {result['question']}\n")
    print(f"Answer: {result['answer']}\n")
    print(f"Sources: {', '.join(result['sources'])}")
```

Similar structure to ingest, but calls `pipeline.query()` instead:

| Step | Code | What Happens |
|------|------|-------------|
| 1 | `ChromaVectorStore(...)` | Open the existing database (must have been ingested before) |
| 2 | `RAGPipeline(...)` | Create a pipeline with the user's LLM model choice |
| 3 | `pipeline.query(...)` | Embed question → search → build prompt → generate answer |
| 4 | `print(...)` | Display the question, answer, and sources |

#### The output format

```bash
$ uv run rag-basics query "What is machine learning?"

Question: What is machine learning?

Answer: Machine learning is a subset of artificial intelligence that enables
systems to automatically learn and improve from experience without being
explicitly programmed. (Sources: machine_learning.txt)

Sources: machine_learning.txt, artificial_intelligence.txt
```

The `', '.join(result['sources'])` turns the list `["ml.txt", "ai.txt"]` into the string `"ml.txt, ai.txt"` — a comma-separated, human-readable format.

## Key Python Concepts Used

| Concept | Where It's Used | One-Sentence Explanation |
|---------|-----------------|--------------------------|
| `argparse.ArgumentParser` | Top-level CLI setup | Python's built-in tool for parsing terminal commands into structured data |
| `add_subparsers` | Creating `ingest` and `query` sub-commands | Like `git commit` vs `git push` — different commands with different options |
| `add_argument` | Defining each option | Declares a field on the "intake form" with name, type, default, and help text |
| `required=True` | `--data-dir` | Makes an argument mandatory (argparse will error if missing) |
| Positional arguments | `question` in query | Arguments without `--` that are required and order-dependent |
| `type=int` | `--chunk-size`, `--top-k` | Tells argparse to convert the string input to an integer |
| `load_dotenv()` | First line of `main()` | Reads `.env` file and sets environment variables (like API keys) |
| `sys.exit(1)` | When no command is given | Exits the program with error code 1 (non-zero = error) |
| `if __name__ == "__main__"` | Bottom of file | Only runs `main()` when the file is executed directly, not imported |
| f-strings | Print statements | Embed variables directly in strings: `f"Ingested {count} chunks"` |
| `', '.join(list)` | Formatting sources | Turns `["a.txt", "b.txt"]` into `"a.txt, b.txt"` |

## Common Mistakes to Avoid

| Mistake | What Goes Wrong | The Fix |
|---------|----------------|---------|
| Forgetting `load_dotenv()` | `OPENAI_API_KEY` not found → all API calls fail | Always call `load_dotenv()` before any pipeline operations |
| Querying before ingesting | Empty vector store → "I don't have enough information" | Always run `ingest` before `query` |
| Using different `--collection` for ingest and query | Query searches the wrong (empty) collection | Use the same collection name for both |
| Using different `--persist-dir` for ingest and query | Query can't find the database | Use the same persist directory for both |
| Forgetting quotes around the question | Shell splits "What is AI?" into separate words → parsing error | Always wrap the question in quotes: `"What is AI?"` |
| Missing `--data-dir` for ingest | Argparse prints an error: "required argument" | Always provide `--data-dir` (it's the only required argument) |
