# Understanding `vector_store.py` — The Filing Cabinet of Your RAG Pipeline

## The Big Picture: What Problem Does This Solve?

You've loaded your documents. You've chunked them into small pieces. You've embedded each chunk into a list of 1536 numbers. Now what?

You need somewhere to **store** all those vectors, and — more importantly — you need a way to **search** through them quickly. When a user asks a question, you embed the question into a vector too, and then find the chunks whose vectors are **closest** to the question's vector.

That's what the vector store does. It's a specialized database designed for one thing: **finding similar vectors fast**.

> **Analogy: The Library with a GPS-Based Filing System**
>
> Imagine a library that doesn't organize books by author or title. Instead, every book has a GPS coordinate based on its **content**. Books about physics are all in the northeast corner. Books about cooking are in the southwest.
>
> When you ask "Tell me about quantum mechanics," the librarian:
> 1. Converts your question into a GPS coordinate (embedding)
> 2. Walks to that spot on the map
> 3. Grabs the 3 closest books (top_k=3)
> 4. Hands them to you
>
> The vector store IS this GPS-organized library. `vector_store.py` gives you two different implementations of this library — **ChromaDB** and **FAISS** — each with different strengths.

---

## Where Does It Fit in the RAG Pipeline?

```
Files on disk → Load → Chunk → Embed → Store → Query → Answer
                                         ^^^
                                    YOU ARE HERE
```

The vector store is the **pivot point** of the entire pipeline. Everything before it (load, chunk, embed) is about **getting data in**. Everything after it (query, answer) is about **getting answers out**.

```
                    INGESTION (one-time setup)
                    ─────────────────────────
                    Load files → Chunk → Embed → STORE
                                                   │
                    RETRIEVAL (every user question) │
                    ────────────────────────────── │
                    User question → Embed → QUERY ─┘→ Top results → LLM → Answer
```

> **Key insight:** You ingest data once, but query it thousands of times. The vector store makes that repeated querying fast.

### 2. `VectorStore` — The Abstract Base Class (The Contract)

```python
class VectorStore(ABC):
    """Abstract base class for vector stores."""

    @abstractmethod
    def add(self, documents: list[Document], embeddings: list[list[float]]) -> None:
        """Add documents with embeddings to the store."""

    @abstractmethod
    def query(self, query_embedding: list[float], top_k: int = 3) -> list[dict]:
        """Query the store for similar documents."""

    @abstractmethod
    def count(self) -> int:
        """Return the number of documents in the store."""
```

This is NOT a real vector store — you can't create an instance of it. It's a **blueprint** that says: *"Any vector store in this project MUST have these three methods."*

> **Analogy: The Restaurant Health Code**
>
> A health code doesn't cook food — it's a set of rules that says "every restaurant MUST have: a hand-washing station, refrigeration, and a fire extinguisher." Any restaurant that doesn't meet these rules can't open.
>
> Similarly, `VectorStore(ABC)` says: "Every vector store MUST have: `add()`, `query()`, and `count()`." Both `ChromaVectorStore` and `FAISSVectorStore` follow this contract, which means **the rest of your code can work with either one without knowing which is being used**.

#### What does `ABC` mean?

`ABC` stands for **Abstract Base Class**. It's Python's way of creating a class that:
1. **Cannot be instantiated** — you can't do `store = VectorStore()` (you'll get an error)
2. **Must be subclassed** — you have to create a child class that implements the abstract methods
3. **Defines a contract** — the `@abstractmethod` decorator marks methods that child classes MUST implement

```python
# This would CRASH:
store = VectorStore()  # TypeError: Can't instantiate abstract class

# This WORKS because ChromaVectorStore implements all abstract methods:
store = ChromaVectorStore()  # OK!
```

#### Why use an abstract class?

> **Analogy: The Power Outlet**
>
> Every electrical outlet in your house follows the same standard (shape, voltage). Because of that standard, you can plug ANY device into ANY outlet. You don't need a special outlet for your laptop and a different one for your lamp.
>
> The `VectorStore` abstract class is that standard. Your `rag_pipeline.py` can work with `ChromaVectorStore` or `FAISSVectorStore` (or any future implementation) without changing a single line of code — because they all have the same `add()`, `query()`, and `count()` methods.

#### The three methods every vector store must have

| Method | Purpose | Analogy |
|--------|---------|---------|
| `add(documents, embeddings)` | Store documents and their vectors | Putting books on the shelves |
| `query(query_embedding, top_k)` | Find the most similar documents | Asking the librarian for recommendations |
| `count()` | How many documents are stored | Counting books in the library |

### 3. `ChromaVectorStore` — The Full-Service Solution

ChromaDB is like a **managed filing service** — it handles storage, indexing, searching, persistence, and metadata filtering all in one package.

#### The Constructor: Setting Up the Database

```python
class ChromaVectorStore(VectorStore):
    """ChromaDB-based vector store with built-in persistence and metadata support."""

    def __init__(self, collection_name: str = "default", persist_dir: str = "./vectordb_data/chromadb"):
        os.makedirs(persist_dir, exist_ok=True)
        self.client = chromadb.PersistentClient(path=persist_dir)
        self.collection = self.client.get_or_create_collection(
            name=collection_name,
            metadata={"hnsw:space": "cosine"},
        )
```

> **Analogy: Opening a Filing Office**
>
> 1. **`os.makedirs(persist_dir, exist_ok=True)`** — Build the office building (create the directory if it doesn't exist). `exist_ok=True` means "don't crash if the building is already there."
>
> 2. **`chromadb.PersistentClient(path=persist_dir)`** — Hire a filing manager who saves everything to that office. "Persistent" means data survives when the program ends (unlike in-memory storage).
>
> 3. **`get_or_create_collection(name=...)`** — Open a specific filing cabinet (or create it if it's new). A "collection" is ChromaDB's word for a group of related vectors — like a drawer in the cabinet.
>
> 4. **`metadata={"hnsw:space": "cosine"}`** — Tell the filing manager: "When comparing items, use cosine similarity." This determines HOW the system measures "closeness" between vectors.

##### What is cosine similarity?

Cosine similarity measures the **angle** between two vectors, not the distance. Two vectors pointing in the same direction have a similarity of 1.0 (identical meaning). Two vectors pointing in opposite directions have -1.0. Perpendicular vectors have 0.0.

```
Similarity = 1.0  → Identical meaning     ("dog" vs "puppy")
Similarity = 0.8  → Very similar           ("dog" vs "pet")
Similarity = 0.3  → Somewhat related       ("dog" vs "animal")
Similarity = 0.0  → Unrelated              ("dog" vs "algebra")
```

> **Analogy: Compass Directions**
>
> Imagine two people each pointing in a direction. Cosine similarity measures how close their pointing directions are:
> - Both pointing north → similarity = 1.0 (same direction)
> - One north, one northeast → similarity = 0.7 (close)
> - One north, one east → similarity = 0.0 (perpendicular)
> - One north, one south → similarity = -1.0 (opposite)

##### What is HNSW?

`hnsw:space` refers to **Hierarchical Navigable Small World** — an algorithm that makes vector search fast. You don't need to understand the algorithm, just know that it's an **index** (like the index at the back of a textbook) that lets ChromaDB quickly find similar vectors without comparing against every single vector in the store.

> **Analogy:** Without HNSW, finding a similar book means walking through the entire library, checking every single book. With HNSW, you have a smart index card system that says "books about AI are in aisle 7, shelf 3" — you jump directly to the right neighborhood.

#### The `add()` Method: Storing Documents

```python
def add(self, documents: list[Document], embeddings: list[list[float]]) -> None:
    """Add documents with embeddings to ChromaDB."""
    ids = []
    texts = []
    metadatas = []

    for doc in documents:
        doc_id = hashlib.md5(doc.content.encode("utf-8")).hexdigest()
        ids.append(doc_id)
        texts.append(doc.content)
        metadatas.append(doc.metadata)

    self.collection.add(
        ids=ids,
        embeddings=embeddings,
        documents=texts,
        metadatas=metadatas,
    )
```

> **Analogy: Filing a Stack of Documents**
>
> For each document in the pile:
> 1. **Stamp a unique ID on it** — like a fingerprint based on its content (`hashlib.md5`)
> 2. **Write down the text** — the actual content that will be returned in search results
> 3. **Attach the metadata label** — source file, chunk index, etc.
>
> Then hand the whole batch to the filing manager (ChromaDB), along with the embeddings. ChromaDB stores everything together and builds its search index.

##### Why `hashlib.md5`?

```python
doc_id = hashlib.md5(doc.content.encode("utf-8")).hexdigest()
```

This line deserves a closer look. It creates a **unique ID** for each document based on its content.

| Part | What It Does |
|------|-------------|
| `doc.content.encode("utf-8")` | Converts the text string to bytes (MD5 needs bytes, not strings) |
| `hashlib.md5(...)` | Runs the MD5 hashing algorithm on those bytes |
| `.hexdigest()` | Converts the hash to a readable hex string like `"a1b2c3d4e5f6..."` |

> **Analogy: A Fingerprint Scanner**
>
> MD5 takes ANY text (short or long) and produces a fixed-length "fingerprint" — a 32-character hex string. The same text always produces the same fingerprint. Different texts produce different fingerprints (with extremely rare exceptions called "collisions").
>
> ```python
> "Machine learning is great"  →  "7f3b9c2e1a4d..."
> "Machine learning is great"  →  "7f3b9c2e1a4d..."   ← Same! (deterministic)
> "Machine learning is Good"   →  "e8a1f4c9d2b7..."   ← Different! (even a tiny change)
> ```

Why not just use a counter (1, 2, 3, ...)? Because if you run the pipeline twice, a counter would create duplicates. With content-based hashing, the same chunk always gets the same ID, so ChromaDB knows it's a duplicate and can handle it.

##### What ChromaDB stores

After `collection.add()`, ChromaDB internally stores a table like this:

| ID (hash) | Embedding (vector) | Document (text) | Metadata |
|-----------|-------------------|-----------------|----------|
| `7f3b9c...` | `[0.023, -0.041, ...]` | `"Machine learning is..."` | `{source: "ml.txt", chunk_index: 0}` |
| `e8a1f4...` | `[0.078, 0.015, ...]` | `"Python programming..."` | `{source: "py.txt", chunk_index: 0}` |
| `a2c4d6...` | `[-0.012, 0.093, ...]` | `"Cloud computing..."` | `{source: "cloud.txt", chunk_index: 0}` |

Everything is tied together — vector + text + metadata — so when a search finds a matching vector, it can return the original text and all its traceability info.

#### The `query()` Method: Searching for Similar Documents

```python
def query(self, query_embedding: list[float], top_k: int = 3, where: dict | None = None) -> list[dict]:
    """Query ChromaDB for similar documents."""
    kwargs = {
        "query_embeddings": [query_embedding],
        "n_results": top_k,
        "include": ["documents", "metadatas", "distances"],
    }
    if where:
        kwargs["where"] = where

    results = self.collection.query(**kwargs)

    output = []
    for i in range(len(results["documents"][0])):
        output.append({
            "content": results["documents"][0][i],
            "metadata": results["metadatas"][0][i],
            "similarity": 1 - results["distances"][0][i],
        })
    return output
```

This is the **heart of RAG retrieval** — the method that actually finds relevant chunks when a user asks a question.

> **Analogy: Asking the Librarian**
>
> 1. You walk up to the librarian with your question converted to a GPS coordinate (`query_embedding`)
> 2. You say "Find me the 3 closest books" (`top_k=3`)
> 3. Optionally, you add a filter: "Only look in the science section" (`where={"source": "science.txt"}`)
> 4. The librarian scans the shelves, finds the 3 closest matches, and tells you how close each one is
> 5. You get back a list: the book text, its label, and a similarity score

##### Building the query step by step

**Step 1: Prepare the search request**
```python
kwargs = {
    "query_embeddings": [query_embedding],   # The question as a vector
    "n_results": top_k,                      # How many results to return
    "include": ["documents", "metadatas", "distances"],  # What info to include
}
```

- `query_embeddings` is wrapped in a list `[...]` because ChromaDB supports multiple queries at once (we're doing just one)
- `include` specifies what to return. Without this, you'd only get IDs

**Step 2: Add optional filtering**
```python
if where:
    kwargs["where"] = where
```

The `where` parameter lets you filter by metadata BEFORE searching. For example:

```python
# Only search within documents from "machine_learning.txt"
results = store.query(
    query_embedding=vec,
    top_k=3,
    where={"source": "machine_learning.txt"}
)
```

> **Analogy:** Instead of searching the entire library, you tell the librarian: "Only look in aisle 5." This narrows down the search and makes it faster and more relevant.

**Step 3: Execute the search**
```python
results = self.collection.query(**kwargs)
```

The `**kwargs` syntax unpacks the dictionary into keyword arguments. It's equivalent to:
```python
results = self.collection.query(
    query_embeddings=[query_embedding],
    n_results=top_k,
    include=["documents", "metadatas", "distances"],
)
```

**Step 4: Format the results**
```python
output = []
for i in range(len(results["documents"][0])):
    output.append({
        "content": results["documents"][0][i],
        "metadata": results["metadatas"][0][i],
        "similarity": 1 - results["distances"][0][i],
    })
return output
```

ChromaDB returns results in a nested structure (lists of lists, because it supports batch queries). We flatten it into a simple list of dictionaries.

The key transformation: **`similarity = 1 - distance`**. ChromaDB returns **distances** (lower = more similar), but it's more intuitive to work with **similarity** (higher = more similar).

```
Distance = 0.1  →  Similarity = 0.9  →  Very similar!
Distance = 0.8  →  Similarity = 0.2  →  Not very similar
```

##### What the output looks like

```python
[
    {
        "content": "Machine learning is a subset of AI that...",
        "metadata": {"source": "ml.txt", "chunk_index": 2},
        "similarity": 0.89
    },
    {
        "content": "Deep learning uses neural networks to...",
        "metadata": {"source": "ai.txt", "chunk_index": 5},
        "similarity": 0.82
    },
    {
        "content": "AI systems can process natural language...",
        "metadata": {"source": "ai.txt", "chunk_index": 1},
        "similarity": 0.76

]
```

Ordered from most similar to least similar. Each result carries its original text, metadata for traceability, and a similarity score.

### 4. `FAISSVectorStore` — The DIY High-Performance Solution

FAISS (Facebook AI Similarity Search) is like buying a **raw engine** instead of a complete car. It's incredibly fast at vector search, but it doesn't manage metadata, persistence, or anything else — you have to handle all of that yourself.

> **Analogy: ChromaDB vs FAISS**
>
> | | **ChromaDB** | **FAISS** |
> |--|---|---|
> | **What it is** | A full-service filing office | A raw filing cabinet |
> | **Search** | Built-in | Built-in (and faster) |
> | **Metadata storage** | Built-in | You manage it yourself |
> | **Persistence (saving to disk)** | Built-in | You manage it yourself |
> | **Filtering** | Built-in (`where` parameter) | You manage it yourself (`filter_fn`) |
> | **Best for** | Simplicity, prototyping | Performance, large-scale |
> | **Analogy** | A restaurant (sit down, order, eat) | A grocery store (buy ingredients, cook yourself) |

#### The Constructor: Setting Up FAISS

```python
class FAISSVectorStore(VectorStore):
    """FAISS-based vector store with manual metadata management and persistence."""

    def __init__(self, dimension: int = 1536, persist_dir: str | None = None):
        self.dimension = dimension
        self.persist_dir = persist_dir
        self.index = faiss.IndexFlatIP(dimension)
        self.doc_store: list[dict] = []

        if persist_dir:
            self._load()
```

##### Line-by-line breakdown

| Line | What It Does | Why |
|------|-------------|-----|
| `self.dimension = dimension` | Stores the vector size (1536 for `text-embedding-3-small`) | FAISS needs to know vector dimensions upfront |
| `self.persist_dir = persist_dir` | Where to save data on disk (or `None` for in-memory only) | Optional persistence — set to `None` if you don't need to save |
| `self.index = faiss.IndexFlatIP(dimension)` | Creates the FAISS index for vector search | The actual search engine — `IP` means "Inner Product" (similarity metric) |
| `self.doc_store: list[dict] = []` | A plain Python list to store document text and metadata | FAISS only stores vectors — we need our own side-storage for text/metadata |
| `if persist_dir: self._load()` | Load previously saved data from disk | If we saved data before, reload it instead of starting empty |

##### What is `IndexFlatIP`?

```python
self.index = faiss.IndexFlatIP(dimension)
```

This creates a FAISS index that uses **Inner Product** (IP) similarity to compare vectors.

- `Flat` = Brute-force search (checks every single vector — exact results, but slower for very large datasets)
- `IP` = Inner Product similarity metric (after normalization, this is equivalent to cosine similarity)

> **Analogy:** `Flat` means the librarian personally checks every book in the library. This gives perfect results but takes longer as the library grows. For millions of books, you'd use a fancier index (like `IndexIVFFlat`) that builds neighborhoods for faster approximate search.

##### The Two-Part Storage System

FAISS only stores **vectors**. It has no concept of text or metadata. So we use two parallel storage systems:

```
FAISS Index (self.index):        Doc Store (self.doc_store):
┌────────────────────────┐       ┌────────────────────────────────┐
│ Index 0: [0.023, ...]  │  ←──► │ {"content": "ML is...",        │
│ Index 1: [0.078, ...]  │  ←──► │  "metadata": {source: "ml"}}  │
│ Index 2: [-0.012, ...] │  ←──► │ {"content": "Python...",       │
└────────────────────────┘       │  "metadata": {source: "py"}}  │
                                 └────────────────────────────────┘
```

When FAISS finds that "Index 1 is the most similar," we look up `self.doc_store[1]` to get the actual text and metadata. The indices MUST stay in sync — item 0 in the FAISS index corresponds to item 0 in the doc store.

> **Analogy: A Parking Lot and a Guest List**
>
> FAISS is like a parking lot where cars (vectors) are parked in numbered spots. The `doc_store` is a guest list that says "Spot 0 = John's red Tesla, Spot 1 = Sarah's blue Honda." When you find a car by its appearance (vector search), you check the guest list to learn who it belongs to (metadata).

#### The `add()` Method: Storing Documents in FAISS

```python
def add(self, documents: list[Document], embeddings: list[list[float]]) -> None:
    """Add documents with embeddings to FAISS."""
    vectors = np.array(embeddings, dtype=np.float32)
    faiss.normalize_L2(vectors)
    self.index.add(vectors)

    for doc in documents:
        self.doc_store.append({"content": doc.content, "metadata": doc.metadata})

    if self.persist_dir:
        self._save()
```

##### Line-by-line breakdown

**Step 1: Convert to numpy array**
```python
vectors = np.array(embeddings, dtype=np.float32)
```

FAISS is written in C++ for speed. It doesn't understand Python lists — it needs **numpy arrays** (a highly optimized numeric data structure). `float32` means 32-bit floating-point numbers (good enough precision, saves memory compared to 64-bit).

> **Analogy:** FAISS speaks C++. Our embeddings are in Python. `np.array()` is the translator between the two languages.

**Step 2: Normalize the vectors**
```python
faiss.normalize_L2(vectors)
```

This scales every vector to have a **length of 1.0** (unit vector). After normalization, the Inner Product between two vectors equals their cosine similarity. This is a mathematical trick that lets us use the faster Inner Product calculation while getting cosine similarity results.

> **Analogy:** Imagine comparing the directions two people are pointing. One person has a very long arm and one has a short arm. If you measure the distance to their fingertips, the long arm always "wins." Normalization makes both arms the same length, so only the **direction** matters — which is what we care about (meaning, not magnitude).

```
Before normalization: [3.0, 4.0]    (length = 5.0)
After normalization:  [0.6, 0.8]    (length = 1.0)
```

**Step 3: Add to the FAISS index**
```python
self.index.add(vectors)
```

This puts the vectors into the FAISS index. Internally, FAISS stores them in an optimized data structure for fast searching.

**Step 4: Store the text and metadata separately**
```python
for doc in documents:
    self.doc_store.append({"content": doc.content, "metadata": doc.metadata})
```

Since FAISS only stores vectors, we maintain a parallel list of document content and metadata. The order matches the FAISS index.

**Step 5: Save to disk (if persistence is enabled)**
```python
if self.persist_dir:
    self._save()
```

Optional auto-save after every `add()` call.

#### The `query()` Method: Searching FAISS

```python
def query(
    self, query_embedding: list[float], top_k: int = 3, filter_fn: Callable | None = None
) -> list[dict]:
    """Search FAISS for similar documents."""
    query_vec = np.array([query_embedding], dtype=np.float32)
    faiss.normalize_L2(query_vec)

    search_k = top_k * 3 if filter_fn else top_k
    scores, indices = self.index.search(query_vec, min(search_k, self.index.ntotal))

    results = []
    for score, idx in zip(scores[0], indices[0]):
        if idx == -1:
            continue
        entry = self.doc_store[idx]
        if filter_fn and not filter_fn(entry["metadata"]):
            continue
        results.append({
            "content": entry["content"],
            "metadata": entry["metadata"],
            "similarity": float(score),
        })
        if len(results) >= top_k:
            break
    return results
```

This is more complex than ChromaDB's query because we're managing filtering manually. Let's break it down:

**Step 1: Prepare the query vector**
```python
query_vec = np.array([query_embedding], dtype=np.float32)
faiss.normalize_L2(query_vec)
```

Same conversion + normalization as in `add()`. The query vector MUST be normalized the same way as the stored vectors, otherwise the similarity scores would be meaningless.

> **Important:** Both the stored vectors AND the query vector must be normalized. It's like both people must have the same arm length for the direction comparison to work.

**Step 2: Over-fetch when filtering**
```python
search_k = top_k * 3 if filter_fn else top_k
```

This is a clever trick. If we have a filter function, some results will be filtered out. If we only fetch `top_k=3` results and 2 get filtered, we'd only have 1 result. So we fetch 3x as many and then filter down to `top_k`.

> **Analogy:** You ask the librarian for 3 science books. But the librarian doesn't know which books are science books until they check (that's the filter). So they grab 9 books from the closest shelf, check which ones are science, and hand you the first 3 that match.

**Step 3: Execute the FAISS search**
```python
scores, indices = self.index.search(query_vec, min(search_k, self.index.ntotal))
```

FAISS returns two parallel arrays:
- `scores` — similarity scores (higher = more similar, since we used Inner Product)
- `indices` — positions in the index (which we use to look up `doc_store`)

`min(search_k, self.index.ntotal)` prevents asking for more results than exist in the index.

```python
scores  = [[0.92, 0.85, 0.78, 0.71, ...]]   # How similar each result is
indices = [[   5,   12,    3,    8, ...]]     # Which position in the index
#              ↓     ↓     ↓     ↓
#         doc_store[5] doc_store[12] ...      # Look up the actual documents
```

**Step 4: Filter and collect results**
```python
for score, idx in zip(scores[0], indices[0]):
    if idx == -1:                                    # FAISS returns -1 for padding
        continue
    entry = self.doc_store[idx]                      # Look up text + metadata
    if filter_fn and not filter_fn(entry["metadata"]):  # Apply custom filter
        continue
    results.append({...})
    if len(results) >= top_k:                        # Stop when we have enough
        break
```

- `idx == -1` — FAISS returns `-1` as a placeholder when there aren't enough results. We skip these.
- `filter_fn` — an optional function you pass in to filter results by metadata.
- We collect until we hit `top_k` results, then stop.

##### How filtering works (ChromaDB vs FAISS)

ChromaDB has built-in filtering with a `where` dictionary:
```python
# ChromaDB: declarative filtering
store.query(vec, top_k=3, where={"source": "ml.txt"})
```

FAISS requires you to pass a **function** that returns `True` or `False`:
```python
# FAISS: functional filtering
store.query(vec, top_k=3, filter_fn=lambda m: m["source"] == "ml.txt")
```

> **Analogy:** ChromaDB is like telling a search engine "filter by category: science." FAISS is like writing your own filter logic: "for each result, check if category equals science, and only keep it if it does."

#### Persistence: `_save()` and `_load()`

```python
def _save(self) -> None:
    os.makedirs(self.persist_dir, exist_ok=True)
    faiss.write_index(self.index, os.path.join(self.persist_dir, "index.faiss"))
    with open(os.path.join(self.persist_dir, "doc_store.json"), "w") as f:
        json.dump(self.doc_store, f)

def _load(self) -> None:
    index_path = os.path.join(self.persist_dir, "index.faiss")
    store_path = os.path.join(self.persist_dir, "doc_store.json")
    if os.path.exists(index_path) and os.path.exists(store_path):
        self.index = faiss.read_index(index_path)
        with open(store_path) as f:
            self.doc_store = json.load(f)
```

Since FAISS doesn't handle persistence, we do it manually with two files:

| File | What It Stores | Format |
|------|---------------|--------|
| `index.faiss` | All the vectors + the FAISS index structure | Binary (FAISS's native format) |
| `doc_store.json` | All document content and metadata | JSON (human-readable) |

> **Analogy: Saving Your Game**
>
> When you save a video game, it writes your progress to disk. When you load, it restores everything. `_save()` writes the index and doc store to disk. `_load()` reads them back. Without these, you'd lose all your embedded data every time the program stops.

The leading underscore in `_save` and `_load` is a Python convention meaning **"private method — don't call this from outside the class."** They're internal helpers, automatically called by `add()` and `__init__`.

## ChromaDB vs FAISS: Which Should You Use?

| Factor | ChromaDB | FAISS |
|--------|----------|-------|
| **Setup complexity** | Easy — handles everything | More work — you manage metadata and persistence |
| **Search speed** | Fast | Faster (especially at scale) |
| **Metadata filtering** | Built-in (`where` dict) | Manual (`filter_fn` callback) |
| **Persistence** | Automatic | Manual (`_save` / `_load`) |
| **Scalability** | Good for millions of vectors | Excellent for billions of vectors |
| **Dependencies** | `chromadb` package | `faiss-cpu` + `numpy` |
| **Best for** | Beginners, prototyping, small-medium scale | Production, large-scale, performance-critical |

> **Rule of thumb for this project:** Start with ChromaDB. It's simpler, and you can always switch to FAISS later — thanks to the abstract base class, the rest of your code won't need to change.

## Key Python Concepts Used

| Concept | Where It's Used | One-Sentence Explanation |
|---------|-----------------|--------------------------|
| `ABC` + `@abstractmethod` | `VectorStore` base class | Defines a contract that child classes must follow |
| Inheritance | `ChromaVectorStore(VectorStore)` | Child class gets all parent methods and must implement abstract ones |
| `hashlib.md5` | Generating document IDs | Creates a deterministic fingerprint from text content |
| `numpy` arrays | FAISS vector operations | Optimized numeric arrays — much faster than Python lists for math |
| `**kwargs` | ChromaDB query building | Passes a dictionary as keyword arguments to a function |
| `Callable` type hint | FAISS `filter_fn` parameter | Declares that a parameter expects a function as its value |
| Private methods (`_save`, `_load`) | FAISS persistence | Leading underscore convention = "internal use only" |
| `os.makedirs(exist_ok=True)` | Directory creation | Creates a directory (and parents), no error if it exists |
| `zip(scores, indices)` | FAISS result iteration | Pairs up elements from two lists for parallel iteration |
| `json.dump` / `json.load` | FAISS doc store persistence | Serializes Python objects to/from JSON files |

## Common Mistakes to Avoid

| Mistake | What Goes Wrong | The Fix |
|---------|----------------|---------|
| Mixing ChromaDB and FAISS vectors | Vectors stored in one can't be searched in the other | Pick one store and stick with it |
| Forgetting to normalize in FAISS | Similarity scores become meaningless | Always call `faiss.normalize_L2()` on BOTH stored and query vectors |
| Mismatched dimensions | FAISS crashes if query vector size differs from index dimension | Use the same embedding model everywhere |
| Not persisting FAISS data | All data lost when the program ends | Set `persist_dir` or call `_save()` |
| Filtering too aggressively with FAISS | No results returned (all filtered out) | The 3x over-fetch helps, but very strict filters may need higher multiplier |
| Adding the same documents twice (ChromaDB) | Duplicate content in results | MD5 IDs help detect this, but ChromaDB may upsert or error depending on config |

## Recap: The Mental Model

> **Vector Store = The Filing Cabinet of your RAG pipeline.**
>
> It's where your embedded chunks finally find a permanent home. Data goes in through `add()` and comes out through `query()`. The magic is in the search — instead of matching exact keywords, it finds documents with the **most similar meaning** using vector similarity.
>
> You have two options:
> - **ChromaDB** — the full-service option. Stores vectors, text, metadata, handles persistence, supports filtering. Easy to use. Like a managed filing service.
> - **FAISS** — the high-performance option. Blazing fast at vector search, but you manage everything else yourself. Like buying a raw engine and building the car around it.
>
> Both follow the same contract (`VectorStore` ABC), so the rest of your pipeline doesn't care which one you're using — plug and play.
