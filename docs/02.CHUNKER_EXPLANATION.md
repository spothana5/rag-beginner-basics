# Understanding `chunker.py` — The Butcher of Your RAG Pipeline

## The Big Picture: What Problem Does This Solve?

Imagine you just read an entire 300-page textbook. Now someone asks you: *"What does the book say about photosynthesis?"* You wouldn't re-read the entire book — you'd flip to the relevant chapter, find the right paragraph, and read just that section.

That's exactly the problem chunking solves. AI models (LLMs) have a **limited context window** — they can only "read" a certain amount of text at a time. And even if they could read everything, searching through a massive document for a tiny answer is slow and inaccurate. So we **break documents into smaller, meaningful pieces** called "chunks."

> **Analogy: The Pizza Analogy**
>
> You've just made a giant pizza (your document). Nobody eats a whole pizza in one bite — you **slice it** into pieces. But *how* you slice matters:
>
> - **Fixed-size chunks** = Cut the pizza into equal squares with a grid cutter. Fast, simple, but you might cut through a topping (a sentence) awkwardly.
> - **Sentence-based chunks** = Cut along the natural topping boundaries. Each slice has complete toppings (sentences), but slices may vary in size.
> - **Recursive chunks** = Try to cut along the biggest natural boundaries first (between sections), then within sections (paragraphs), then within paragraphs (sentences), and only as a last resort, cut mid-sentence. The smartest approach.
>
> `chunker.py` gives you all three slicing strategies.

---

## Where Does It Fit in the RAG Pipeline?

```
Files on disk → Load → Chunk → Embed → Store → Query → Answer
                        ^^^
                   YOU ARE HERE
```

The chunker sits right after the document loader. It takes the `Document` objects produced by `document_loader.py` and splits each one into **smaller `Document` objects** — each representing a chunk. These chunks are what actually get embedded and stored in the vector database.

> **Why not skip chunking and embed whole documents?**
>
> Three reasons:
> 1. **Embedding models have size limits** — Most models can only process ~512 tokens (~2000 characters) at a time
> 2. **Precision** — If you search a 10,000-character document, the answer might be in one 200-character paragraph. Searching the whole document would dilute that signal with 9,800 characters of irrelevant text
> 3. **Relevance** — Smaller chunks = more precise search results = better answers from the LLM

### 2. `chunk_by_size()` — The Grid Cutter (Fixed-Size Chunking)

```python
def chunk_by_size(doc: Document, chunk_size: int = 500, overlap: int = 50) -> list[Document]:
    """Split document into fixed-size chunks with overlap."""
    text = doc.content
    chunks = []
    start = 0
    chunk_index = 0

    while start < len(text):
        end = start + chunk_size
        chunk_text = text[start:end]

        chunk_metadata = {
            **doc.metadata,
            "chunk_index": chunk_index,
            "chunk_method": "fixed_size",
            "chunk_char_count": len(chunk_text),
        }
        chunks.append(Document(content=chunk_text, metadata=chunk_metadata))

        start += chunk_size - overlap
        chunk_index += 1

    return chunks
```

> **Analogy: Cutting a Ribbon**
>
> Imagine you have a long ribbon (your text) and a ruler. You measure out exactly 500cm, cut. Measure another 500cm, cut. Repeat until you run out of ribbon.
>
> But here's the trick — you **overlap** each cut by 50cm. So the end of one piece and the beginning of the next piece share 50cm of ribbon. Why? Because if an important pattern spans the boundary between two cuts, the overlap ensures it appears completely in at least one chunk.

#### The Parameters

| Parameter | Default | What It Controls |
|-----------|---------|-----------------|
| `doc` | (required) | The Document to split |
| `chunk_size` | `500` | How many characters per chunk |
| `overlap` | `50` | How many characters to repeat between consecutive chunks |

#### Line-by-line breakdown

**Setting up:**
```python
text = doc.content       # Extract the raw text string from the Document
chunks = []              # Empty list to collect our chunks
start = 0               # Position where the current chunk begins
chunk_index = 0          # Counter: chunk 0, chunk 1, chunk 2, ...
```

**The main loop:**
```python
while start < len(text):           # Keep going until we've processed all the text
    end = start + chunk_size       # Where this chunk ends
    chunk_text = text[start:end]   # Slice out the chunk (Python string slicing)
```

> **Analogy: The Sliding Window**
>
> Picture a window frame that's 500 characters wide. You place it at the start of the text and see the first 500 characters. Then you **slide** it forward — but not by a full 500. You slide it by `chunk_size - overlap` = 450 characters. So the window now shows characters 450–950. The 50-character overlap (characters 450–500) appears in both chunks.

```
Text: |-------- 500 chars --------|
       ^chunk 1                    ^

            |-------- 500 chars --------|
            ^chunk 2 (starts 450 chars in — 50 char overlap)
```

**Building metadata:**
```python
chunk_metadata = {
    **doc.metadata,                        # Copy ALL metadata from the parent document
    "chunk_index": chunk_index,            # Which chunk number this is (0, 1, 2, ...)
    "chunk_method": "fixed_size",          # How it was chunked
    "chunk_char_count": len(chunk_text),   # How many characters in this chunk
}
```

The `**doc.metadata` syntax is called **dictionary unpacking**. It copies all key-value pairs from the original document's metadata into the new dictionary.

> **Analogy: Photocopying the Shipping Label**
>
> Remember the shipping label (metadata) from the document loader? When you cut a document into chunks, each chunk gets a **photocopy** of the original label, plus some extra notes stapled on top:
> - "This is chunk #3"
> - "It was created using fixed-size chunking"
> - "It contains 487 characters"
>
> This way, if you later find a great search result, you can trace it back: *"This came from chunk 3 of `machine_learning.txt`, created via fixed-size chunking."*

**Advancing the window:**
```python
start += chunk_size - overlap   # Move forward, but leave room for overlap
chunk_index += 1                # Increment the chunk counter
```

This is the key line. `start` doesn't jump by a full `chunk_size` — it jumps by `chunk_size - overlap`. This is what creates the overlap between consecutive chunks.

#### Visual Example

Given text = `"ABCDEFGHIJKLMNOPQRST"` (20 characters), `chunk_size=8`, `overlap=3`:

```
Step 1: start=0,  end=8   → "ABCDEFGH"       (chars 0-7)
Step 2: start=5,  end=13  → "FGHIJKLM"       (chars 5-12, overlap: "FGH")
Step 3: start=10, end=18  → "KLMNOPQR"       (chars 10-17, overlap: "KLM")
Step 4: start=15, end=20  → "PQRST"          (chars 15-19, overlap: "PQRST"... wait, "PQR")
```

Notice:
- Each chunk starts 5 characters after the previous one (`chunk_size - overlap` = 8 - 3 = 5)
- 3 characters overlap between consecutive chunks
- The last chunk is shorter than `chunk_size` — and that's fine

#### Why Overlap Matters

> **Analogy: The Jigsaw Puzzle**
>
> Imagine cutting a photograph into pieces for a jigsaw puzzle. If you cut exactly along hard lines, a face might be split right down the middle — half in one piece, half in another. Neither piece shows the full face. But if each piece extends slightly into its neighbor's territory (overlap), the full face appears in at least one piece.
>
> In text, a sentence like *"Machine learning is a subset of artificial intelligence"* might land right at the boundary between two chunks. Without overlap, chunk 1 gets *"Machine learning is a subset of"* and chunk 2 gets *"artificial intelligence."* Neither chunk captures the full meaning. With overlap, both chunks contain the complete sentence.

#### Pros and Cons

| Pros | Cons |
|------|------|
| Simple and predictable | Cuts can land mid-word or mid-sentence |
| Every chunk is (roughly) the same size | No awareness of text structure |
| Fast — just string slicing | May split a key idea across chunks |
| Easy to reason about chunk count | Not ideal for text where meaning depends on context |

### 3. `chunk_by_sentences()` — The Natural Boundary Cutter

```python
def chunk_by_sentences(doc: Document, sentences_per_chunk: int = 5, overlap_sentences: int = 1) -> list[Document]:
    """Split document into chunks based on sentence boundaries."""
    sentences = re.split(r"(?<=[.!?])\s+", doc.content)
    sentences = [s.strip() for s in sentences if s.strip()]

    chunks = []
    start = 0
    chunk_index = 0

    while start < len(sentences):
        end = min(start + sentences_per_chunk, len(sentences))
        chunk_text = " ".join(sentences[start:end])

        chunk_metadata = {
            **doc.metadata,
            "chunk_index": chunk_index,
            "chunk_method": "sentence",
            "num_sentences": end - start,
            "chunk_char_count": len(chunk_text),
        }
        chunks.append(Document(content=chunk_text, metadata=chunk_metadata))

        start += sentences_per_chunk - overlap_sentences
        chunk_index += 1

    return chunks
```

> **Analogy: Cutting Along the Dotted Lines**
>
> Instead of blindly cutting every 500 characters (which might cut a sentence in half), this method respects the natural "dotted lines" in text — the sentence boundaries. It groups sentences together (5 at a time by default) and cuts between them. No sentence is ever split in half.

#### The Parameters

| Parameter | Default | What It Controls |
|-----------|---------|-----------------|
| `doc` | (required) | The Document to split |
| `sentences_per_chunk` | `5` | How many sentences to group into each chunk |
| `overlap_sentences` | `1` | How many sentences to repeat between consecutive chunks |

#### Step 1: Splitting Text into Sentences

```python
sentences = re.split(r"(?<=[.!?])\s+", doc.content)
sentences = [s.strip() for s in sentences if s.strip()]
```

This is the trickiest line in the file. Let's break down the regex `(?<=[.!?])\s+`:

| Pattern Part | What It Means |
|-------------|---------------|
| `(?<=...)` | **Lookbehind assertion** — match what comes AFTER this pattern, but don't consume the pattern itself |
| `[.!?]` | A period, exclamation mark, or question mark |
| `\s+` | One or more whitespace characters (spaces, tabs, newlines) |

> **Analogy: The Sentence Detector**
>
> Imagine reading a paragraph aloud. Every time you hit a period, exclamation mark, or question mark followed by a space, that's where one sentence ends and the next begins. The regex does exactly this — it finds those boundaries and splits the text there.

**Example:**

```python
text = "AI is amazing. It learns from data! Can it think? Maybe someday."

sentences = re.split(r"(?<=[.!?])\s+", text)
# Result: ["AI is amazing.", "It learns from data!", "Can it think?", "Maybe someday."]
```

The lookbehind `(?<=...)` is crucial — it splits *after* the punctuation, keeping the `.`, `!`, or `?` attached to the sentence it belongs to. Without the lookbehind, the punctuation would be lost.

**The cleanup line:**

```python
sentences = [s.strip() for s in sentences if s.strip()]
```

This is a **list comprehension** that does two things:
1. `s.strip()` — removes leading/trailing whitespace from each sentence
2. `if s.strip()` — filters out any empty strings (which can happen with extra whitespace in the input)

> **Analogy:** After cutting coupons from a newspaper, you trim the ragged edges (`.strip()`) and throw away any blank scraps (`if s.strip()`).

#### Step 2: Grouping Sentences into Chunks

```python
while start < len(sentences):
    end = min(start + sentences_per_chunk, len(sentences))
    chunk_text = " ".join(sentences[start:end])
```

This works just like `chunk_by_size()`, but instead of sliding a window over *characters*, we slide it over *sentences*.

> **Analogy: Dealing Cards**
>
> Imagine your sentences are a deck of cards. You deal 5 cards (sentences) into a pile (chunk). Then you take back the last card (overlap of 1), and deal 5 more starting from that card. Each pile shares 1 card with the previous pile.

**Visual example** with `sentences_per_chunk=3`, `overlap_sentences=1`:

```
Sentences: [S1, S2, S3, S4, S5, S6, S7, S8]

Chunk 0: [S1, S2, S3]           ← sentences 0-2
Chunk 1: [S3, S4, S5]           ← sentences 2-4 (S3 overlaps)
Chunk 2: [S5, S6, S7]           ← sentences 4-6 (S5 overlaps)
Chunk 3: [S7, S8]               ← sentences 6-7 (S7 overlaps, last chunk is shorter)
```

The advance step is: `start += sentences_per_chunk - overlap_sentences` = 3 - 1 = 2 sentences forward each time.

#### Extra metadata: `num_sentences`

```python
"num_sentences": end - start,
```

Unlike fixed-size chunks, sentence chunks track **how many sentences** they contain. This is useful because the last chunk often has fewer sentences than the others. You might want to filter out very short chunks later.

#### Pros and Cons

| Pros | Cons |
|------|------|
| Never cuts a sentence in half | Chunk sizes vary — some may be very short or very long |
| Each chunk is semantically coherent | Simple regex may fail on abbreviations like "Dr." or "U.S." |
| Easy to understand | Doesn't consider paragraph or section structure |
| Overlap keeps context at boundaries | Long sentences can create unexpectedly large chunks |

### 4. `chunk_recursive()` — The Smart Cutter (Recursive Chunking)

```python
def chunk_recursive(
    doc: Document,
    chunk_size: int = 500,
    overlap: int = 50,
    separators: list[str] | None = None,
) -> list[Document]:
    """Recursively split text using a hierarchy of separators."""
    if separators is None:
        separators = ["\n\n", "\n", ". ", " "]
```

This is the most sophisticated chunking strategy — and the one most commonly used in production RAG systems. It's also the approach used by LangChain's `RecursiveCharacterTextSplitter`.

> **Analogy: The Skilled Butcher**
>
> Imagine a butcher who needs to cut a large piece of meat into portions no bigger than 500 grams. A bad butcher just hacks through it every 500g regardless of structure. A good butcher:
>
> 1. First tries to separate along the **major joints** (paragraph breaks `\n\n`)
> 2. If a piece is still too big, cuts along **smaller joints** (line breaks `\n`)
> 3. If still too big, cuts between **sentences** (`. `)
> 4. As a last resort, cuts between **words** (` `)
> 5. If a word itself is too long (very rare), cuts mid-word
>
> This preserves the most natural structure possible. You'd never cut mid-sentence if you could cut between paragraphs instead.

#### The Separator Hierarchy

```python
separators = ["\n\n", "\n", ". ", " "]
```

This list is ordered from **coarsest to finest** — from the biggest natural boundaries to the smallest:

| Priority | Separator | What It Represents | When It's Used |
|----------|-----------|-------------------|----------------|
| 1st (best) | `"\n\n"` | Paragraph break | Try this first — paragraphs are the most natural division |
| 2nd | `"\n"` | Line break | If paragraphs are too big, split at line breaks |
| 3rd | `". "` | Sentence end | If lines are too big, split at sentence boundaries |
| 4th (last resort) | `" "` | Space between words | If sentences are too long, split at word boundaries |

> **Analogy: Splitting a Book**
>
> Think about how you'd divide a book into readable excerpts:
> 1. First, try splitting by **chapters** (biggest divisions)
> 2. If a chapter is too long, split by **sections** within the chapter
> 3. If a section is too long, split by **paragraphs**
> 4. If a paragraph is too long, split by **sentences**
> 5. Only as a last resort, split mid-sentence by **words**
>
> You always try the coarsest split first, and only go finer when you have to.

#### The Inner Function: `_split_text()`

```python
def _split_text(text: str, seps: list[str]) -> list[str]:
    if not seps:
        return [text[i : i + chunk_size] for i in range(0, len(text), chunk_size - overlap)]

    sep = seps[0]
    parts = text.split(sep)

    result = []
    current = ""

    for part in parts:
        candidate = current + sep + part if current else part
        if len(candidate) <= chunk_size:
            current = candidate
        else:
            if current:
                result.append(current)
            if len(part) > chunk_size:
                result.extend(_split_text(part, seps[1:]))
                current = ""
            else:
                current = part

    if current:
        result.append(current)

    return result
```

This is a **nested function** (a function defined inside another function). It's the heart of the recursive logic. Let's trace through it step by step.

> **What does "recursive" mean?**
>
> **Analogy: Russian Nesting Dolls (Matryoshka)**
>
> Recursion means a function calls *itself* to solve a smaller version of the same problem. Like opening a Russian nesting doll — inside each doll is a smaller doll. You keep opening until you reach the tiniest one.
>
> Here, `_split_text()` tries to split text with the coarsest separator. If any piece is still too big, it calls *itself* with the next finer separator. And so on, until it runs out of separators.

#### Tracing through the logic step by step

**Base case — no separators left:**
```python
if not seps:
    return [text[i : i + chunk_size] for i in range(0, len(text), chunk_size - overlap)]
```

If we've exhausted all separators and the text is *still* too big, we fall back to brute-force character slicing (just like `chunk_by_size`). This is the safety net — it guarantees we always produce chunks within the size limit.

**Try the current separator:**
```python
sep = seps[0]            # Take the first (coarsest) separator
parts = text.split(sep)  # Split the text by that separator
```

For example, if `sep = "\n\n"` and the text has 3 paragraphs, `parts` will be a list of 3 strings.

**Accumulate parts into chunks:**
```python
result = []
current = ""

for part in parts:
    candidate = current + sep + part if current else part
    if len(candidate) <= chunk_size:
        current = candidate
    else:
        if current:
            result.append(current)
        if len(part) > chunk_size:
            result.extend(_split_text(part, seps[1:]))
            current = ""
        else:
            current = part
```

> **Analogy: Packing a Suitcase**
>
> Imagine you're packing items (parts) into suitcases (chunks) that can hold a maximum of 500 units:
>
> 1. Pick up the next item
> 2. **"Will it fit in the current suitcase?"** (`len(candidate) <= chunk_size`)
>    - Yes → Put it in (`current = candidate`)
>    - No → Close the current suitcase (`result.append(current)`) and check the item:
>      - **Is the item itself too big for any suitcase?** (`len(part) > chunk_size`)
>        - Yes → Break it into smaller pieces using a finer cutting method (`_split_text(part, seps[1:])`) — this is the **recursion**!
>        - No → Start a new suitcase with this item (`current = part`)

Let's trace through a concrete example:

```
Text: "Paragraph 1 is short.\n\nParagraph 2 is also short.\n\nParagraph 3 is really really long and exceeds the chunk_size limit..."
chunk_size: 60

Step 1: Split by "\n\n" → ["Paragraph 1 is short.", "Paragraph 2 is also short.", "Paragraph 3 is really really long..."]

Step 2: Try to pack:
  - "Paragraph 1 is short." (21 chars) → fits in suitcase → current = "Paragraph 1 is short."
  - candidate = "Paragraph 1 is short.\n\nParagraph 2 is also short." (50 chars) → fits! → current = both paragraphs
  - candidate = current + "\n\n" + Paragraph 3 → too big!
    → Close suitcase with paragraphs 1+2
    → Paragraph 3 alone is still > 60 chars → RECURSE with next separator "\n"
    → If "\n" doesn't help, try ". ", then " "
```

**Don't forget the last suitcase:**
```python
if current:
    result.append(current)
```

After the loop ends, there might be content left in `current` that hasn't been added to `result` yet. This line catches it.

#### Wrapping chunks into Documents

```python
pieces = _split_text(doc.content, separators)
chunks = []
for i, piece in enumerate(pieces):
    chunk_metadata = {
        **doc.metadata,
        "chunk_index": i,
        "chunk_method": "recursive",
        "chunk_char_count": len(piece),
    }
    chunks.append(Document(content=piece.strip(), metadata=chunk_metadata))

return chunks
```

After `_split_text()` returns a list of text pieces, we wrap each one in a `Document` with metadata — same pattern as the other chunking methods. The `piece.strip()` removes any leading/trailing whitespace that may have accumulated during splitting.

`enumerate(pieces)` gives us both the index `i` and the piece itself — so we don't need a separate `chunk_index` counter.

#### Pros and Cons

| Pros | Cons |
|------|------|
| Preserves natural text structure (paragraphs > sentences > words) | More complex to understand and debug |
| Produces semantically meaningful chunks | Chunk sizes are less uniform |
| Most similar to how a human would split text | No overlap between chunks (unlike the other two methods) |
| Industry standard — used by LangChain, LlamaIndex, etc. | The separator list may need tuning for specific content types |

## Comparing All Three Strategies Side by Side

Let's see how each method handles the same input:

```
Input text (180 chars):
```

```
Chunk 0: "Machine learning is a branch of AI. It uses algorithms to learn patterns.\n\nDe"
```

Clean sentence boundaries! But notice chunk sizes vary (73, 63, 68, 70 chars).

### `chunk_recursive(chunk_size=80)`

```
Chunk 0: "Machine learning is a branch of AI. It uses algorithms to learn patterns."
Chunk 1: "Deep learning is a subset. It uses neural networks with many layers."
Chunk 2: "NLP processes human language."
```

Split at paragraph boundaries (`\n\n`) first — the most natural division. Each chunk is a complete paragraph.

## The Metadata Pattern: Why Every Chunk Remembers Its Parents

All three methods use this pattern:

```python
chunk_metadata = {
    **doc.metadata,            # Inherit everything from the parent
    "chunk_index": ...,        # Add chunk-specific info
    "chunk_method": "...",
    "chunk_char_count": ...,
}
```

> **Analogy: A Family Tree on Every Package**
>
> When a factory receives raw material (the original document), it stamps the material's origin info on it: "Came from Supplier X, batch #42." When it cuts the material into pieces, each piece gets a copy of that stamp, plus its own: "This is piece 3 of 7, cut using method Y."
>
> Later, when a customer (the query engine) finds a useful piece, they can trace it all the way back: *"This answer came from chunk 2, which was created by sentence-based chunking, from the file `machine_learning.txt` located at `/data/sample_texts/machine_learning.txt`."*

This traceability is critical in production RAG systems — you need to cite your sources.

## Which Strategy Should You Use?

| Situation | Best Strategy | Why |
|-----------|--------------|-----|
| Quick prototype | `chunk_by_size` | Simplest, fewest decisions to make |
| Text with clear sentences | `chunk_by_sentences` | Respects natural boundaries |
| Production system | `chunk_recursive` | Best balance of structure and size control |
| Structured text (code, markdown) | `chunk_recursive` with custom separators | You can tailor the separator hierarchy |

For this beginner project, all three are available so you can experiment and see the difference in search results.

## Key Python Concepts Used

| Concept | Where It's Used | One-Sentence Explanation |
|---------|-----------------|--------------------------|
| `re.split()` | Sentence splitting | Splits a string using a regex pattern instead of a fixed string |
| Lookbehind `(?<=...)` | Sentence regex | Matches a position *after* a pattern without consuming it |
| List comprehension | Cleaning sentences | Compact syntax for filtering and transforming lists |
| `**dict` (unpacking) | Metadata inheritance | Copies all key-value pairs from one dict into another |
| Nested functions | `_split_text` inside `chunk_recursive` | A function defined inside another function, with access to the outer function's variables |
| Recursion | `_split_text` calling itself | A function solving a problem by calling itself with a smaller input |
| `enumerate()` | Wrapping recursive chunks | Loops over a list while also tracking the index |
| Type hint `list[str] \| None` | `separators` parameter | Union type — the parameter accepts either a list of strings or `None` |

## Common Mistakes to Avoid

| Mistake | What Goes Wrong | The Fix |
|---------|----------------|---------|
| Setting `overlap >= chunk_size` | Infinite loop — the window never advances | Ensure `overlap` is always less than `chunk_size` |
| Overlap of 0 | No context shared at boundaries — key ideas may be split | Use at least some overlap (10-20% of chunk_size) |
| Chunk size too small | Chunks lose context — a 50-char chunk has no meaning on its own | 200-1000 chars is a typical range |
| Chunk size too large | Defeats the purpose of chunking — search results are too broad | Keep under the embedding model's token limit |
| Forgetting to strip whitespace | Chunks start/end with `\n` or spaces — wastes embedding capacity | Use `.strip()` on chunk text |
| Not inheriting parent metadata | Chunks lose their source info — can't trace answers back to files | Always use `**doc.metadata` to copy parent metadata |

## Recap: The Mental Model

> **Chunker = The Butcher of your RAG pipeline.**
>
> It takes whole documents (large pieces of meat) and cuts them into portions small enough to be useful (chunks). *How* it cuts matters enormously:
>
> - **`chunk_by_size`**: The grid cutter — equal slices, fast, but may cut through the good parts
> - **`chunk_by_sentences`**: The dotted-line cutter — respects sentences, but sizes vary
> - **`chunk_recursive`**: The skilled butcher — cuts at the most natural boundaries first, producing the highest-quality chunks
>
> The output is always the same: a list of smaller `Document` objects, each carrying its content and a metadata label tracing it back to its origin. These chunks are now ready to be embedded into vectors and stored for searching.
