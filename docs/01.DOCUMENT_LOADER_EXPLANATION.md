# Understanding `document_loader.py` — The Front Door of Your RAG Pipeline

## The Big Picture: What Problem Does This Solve?

Imagine you're building a **super-smart librarian** that can answer questions by reading through books. Before the librarian can do *anything*, someone has to take those books off the shelves and open them up. That's what `document_loader.py` does.

> **Analogy: The Mail Room**
>
> Think of your RAG pipeline as a big office building:
> - **Document Loader** = The mail room. Packages (files) arrive in different formats (envelopes, boxes, tubes). The mail room opens everything, organizes it onto standard trays, and passes it along.
> - **Chunker** = The person who breaks long memos into sticky-note-sized summaries.
> - **Embedder** = The person who translates each sticky note into a secret numeric code for the filing system.
> - **Vector Store** = The filing cabinet that stores everything by those codes.
> - **Query Engine** = The assistant who finds the right sticky notes when you ask a question.
>
> Without the mail room, nothing else works. The mail room doesn't care what's *inside* the packages — it just opens them and puts everything into a standard format.

---

## Where Does It Fit in the RAG Pipeline?

```
Files on disk → Load → Chunk → Embed → Store → Query → Answer
                 ^^^
            YOU ARE HERE
```

The document loader is **Step 1**. Its sole responsibility: take raw files (`.txt` or `.pdf`) and convert them into uniform Python objects called `Document`s. Every module downstream speaks the "Document language" — so this loader is the translator between the messy real world (different file formats, encodings, locations) and the clean internal world of your pipeline.

### 2. The `Document` Dataclass — Your Universal Envelope

```python
@dataclass
class Document:
    """A document with content and metadata."""

    content: str
    metadata: dict = field(default_factory=dict)
```

This is the **most important thing in the entire project**. Every single module uses `Document`.

> **Analogy: The Standardized Shipping Box**
>
> Imagine Amazon receives products from thousands of suppliers — some come in bags, some in crates, some wrapped in newspaper. Amazon puts everything into standardized brown boxes with a shipping label.
>
> - **`content`** = The product inside the box (the actual text from your file)
> - **`metadata`** = The shipping label (information *about* the content — where it came from, how big it is, what type of file it was)
>
> Once everything is in a standard box, the warehouse doesn't care whether the original product came in a bag or a crate. Same idea here — once your text is in a `Document`, the rest of the pipeline doesn't care whether it came from a `.txt` or a `.pdf`.

#### What exactly is a `@dataclass`?

A dataclass is Python's way of saying: *"I want a class whose main job is to hold data. Please generate the boring parts for me."*

Without `@dataclass`, you'd write this:

```python
class Document:
    def __init__(self, content, metadata=None):
        self.content = content
        self.metadata = metadata if metadata is not None else {}

    def __repr__(self):
        return f"Document(content='{self.content}', metadata={self.metadata})"

    def __eq__(self, other):
        return self.content == other.content and self.metadata == other.metadata
```

With `@dataclass`, Python auto-generates `__init__`, `__repr__`, and `__eq__` for you. You just declare the fields and move on.

#### The Mutable Default Trap: Why `field(default_factory=dict)`?

This is a **classic Python gotcha** that trips up even experienced developers.

> **Analogy: The Shared Notebook**
>
> Imagine a classroom where the teacher says, "Everyone gets the default notebook." But instead of giving each student a *new* notebook, she gives them all the *same one*. When Student A writes in it, Student B sees those notes too!
>
> That's what happens with `metadata: dict = {}` — Python creates **one dict object** and shares it across all Document instances.

```python
# BAD — shared mutable default
doc1 = Document(content="hello")
doc2 = Document(content="world")
doc1.metadata["source"] = "file1.txt"
print(doc2.metadata)  # {'source': 'file1.txt'} — BUG! doc2 was affected!
```

`field(default_factory=dict)` fixes this by telling Python: *"Call `dict()` to create a brand-new empty dictionary for each instance."* Each student now gets their own notebook.

```python
# GOOD — each Document gets its own fresh dict
doc1 = Document(content="hello")
doc2 = Document(content="world")
doc1.metadata["source"] = "file1.txt"
print(doc2.metadata)  # {} — doc2 is unaffected. Correct!
```

**Rule of thumb:** Whenever your default value is something mutable (dict, list, set), always use `field(default_factory=...)`.

#### The Two Fields

| Field | Type | Real-World Equivalent |
|-------|------|-----------------------|
| `content` | `str` | The full text of the document — the "meat" |
| `metadata` | `dict` | The label on the box — file name, path, type, character count |

Why separate them? Because later, when you search your vector store, metadata lets you **filter** (e.g., "only search documents from `machine_learning.txt`") and **trace** answers back to their source (e.g., "This answer came from page 3 of `report.pdf`").

#### The `__repr__` Method — Making Debugging Friendly

```python
def __repr__(self) -> str:
    preview = self.content[:80] + "..." if len(self.content) > 80 else self.content
    return f"Document(content='{preview}', metadata={self.metadata})"
```

> **Analogy:** When you look at a book on a shelf, you see the title on the spine — not the entire book. This method shows you just the first 80 characters as a "spine label" so printing a document doesn't flood your terminal with 5,000 characters.

```python
print(doc)
# Document(content='Artificial Intelligence (AI) refers to the simulation of human...', metadata={'source': 'ai.txt'})
```

### 3. `load_text_file()` — Opening a Text File

```python
def load_text_file(file_path: str) -> Document:
    """Load a text file into a Document."""
    path = Path(file_path)
    content = path.read_text(encoding="utf-8")
    metadata = {
        "source": path.name,
        "file_path": str(path),
        "file_type": path.suffix,
        "char_count": len(content),
    }
    return Document(content=content, metadata=metadata)
```

> **Analogy: Opening a Letter**
>
> 1. You pick up the envelope (`Path(file_path)`)
> 2. You open it and read the letter inside (`path.read_text(encoding="utf-8")`)
> 3. You note down details from the envelope — who sent it, the address, how long the letter is (`metadata = {...}`)
> 4. You clip the letter and the notes together and place them on the standard tray (`return Document(...)`)

#### Line-by-line breakdown

| Line | What It Does | Why |
|------|-------------|-----|
| `path = Path(file_path)` | Converts string `"/some/path/file.txt"` into a `Path` object | Gives us handy methods like `.read_text()`, `.name`, `.suffix` |
| `path.read_text(encoding="utf-8")` | Reads the entire file into a string | `utf-8` is specified explicitly to avoid encoding bugs on Windows |
| `path.name` | Returns `"artificial_intelligence.txt"` | The filename only (no directory path) |
| `path.suffix` | Returns `".txt"` | Just the file extension |
| `len(content)` | Character count of the loaded text | Useful for debugging and analytics |
| `return Document(...)` | Wraps everything into a standard `Document` | Now it's ready for the rest of the pipeline |

#### Quick `Path` cheat sheet

```python
path = Path("/Users/spothana/data/sample_texts/artificial_intelligence.txt")

path.name    # "artificial_intelligence.txt"   (full filename)
path.suffix  # ".txt"                          (extension only)
path.stem    # "artificial_intelligence"        (filename without extension)
path.parent  # Path("/Users/spothana/data/sample_texts")  (the folder)
```

### 4. `load_pdf_file()` — Opening a PDF File

```python
def load_pdf_file(file_path: str) -> Document:
    """Load a PDF file into a Document."""
    path = Path(file_path)
    reader = PdfReader(str(path))

    pages_text = []
    for page in reader.pages:
        text = page.extract_text()
        if text:
            pages_text.append(text)

    content = "\n\n".join(pages_text)
    metadata = {
        "source": path.name,
        "file_path": str(path),
        "file_type": ".pdf",
        "num_pages": len(reader.pages),
        "char_count": len(content),
    }
    return Document(content=content, metadata=metadata)
```

> **Analogy: The Multi-Page Fax**
>
> A PDF is like a fax that arrives as multiple separate pages. You can't just "read" the whole thing at once — you have to go page by page:
> 1. Open the fax cover (`PdfReader(...)`)
> 2. Go through each page, read what's on it, and jot it down (`for page in reader.pages`)
> 3. Skip any blank pages (`if text:`)
> 4. Staple all your notes together into one document (`"\n\n".join(pages_text)`)
> 5. Write down how many pages there were on the cover sheet (`num_pages`)

#### Key differences from text loading

| Aspect | Text Files | PDF Files |
|--------|-----------|-----------|
| How we read | `path.read_text()` — one shot | Page-by-page loop with `PdfReader` |
| Extra metadata | — | `num_pages` (how many pages the PDF has) |
| Blank content | Unlikely | Common (image-only pages) — handled by `if text:` |
| Library needed | None (built-in) | `pypdf` (third-party) |

#### Limitations to know about

`pypdf` extracts **embedded text** from PDFs. It does **not** perform OCR (Optical Character Recognition).

> **Analogy:** Imagine photocopying a page from a book. The photocopy *looks* like text, but a computer sees it as a picture. `pypdf` can only read PDFs where the text is *actually text* (selectable/highlightable), not pictures of text. For scanned documents, you'd need a tool like `pytesseract` that can "see" and interpret images.

- PDF with selectable text → Works great
- Scanned PDF (image of text) → Returns empty strings

### 5. `load_directory()` — Loading a Whole Folder at Once

```python
def load_directory(dir_path: str, pattern: str = "*.txt") -> list[Document]:
    """Load all matching files from a directory."""
    path = Path(dir_path)
    documents = []
    for file_path in sorted(path.glob(pattern)):
        if file_path.suffix == ".pdf":
            doc = load_pdf_file(str(file_path))
        else:
            doc = load_text_file(str(file_path))
        documents.append(doc)
    return documents
```

> **Analogy: The Mail Room's Batch Processing**
>
> Instead of opening one letter at a time, the mail room worker grabs **all envelopes from a particular bin** (the directory), sorts them alphabetically, and processes each one:
> - If it's a PDF-shaped package → use the PDF opener
> - If it's anything else → use the regular letter opener
>
> At the end, everything is stacked neatly on the standard tray as a list of `Document`s.

#### How each part works

1. **`path.glob(pattern)`** — Like telling the mail room: "Give me all envelopes that match this description."

   | Pattern | What It Matches |
   |---------|-----------------|
   | `"*.txt"` | All `.txt` files in the folder |
   | `"*.pdf"` | All `.pdf` files in the folder |
   | `"*.*"` | All files with any extension |
   | `"**/*.txt"` | All `.txt` files, including subfolders (recursive) |

2. **`sorted(...)`** — Processes files in alphabetical order. Without this, the order depends on the operating system and is unpredictable. Sorting makes your results consistent and reproducible.

3. **The `if/else` dispatch** — A simple routing rule: PDFs go to the PDF handler, everything else goes to the text handler. This is called the **dispatcher pattern**.

4. **Returns `list[Document]`** — A list of all loaded documents, ready for the next step (chunking).

## Common Mistakes to Avoid

| Mistake | What Goes Wrong | The Fix |
|---------|----------------|---------|
| Forgetting `encoding="utf-8"` | `UnicodeDecodeError` on files with accented characters or emoji | Always pass `encoding="utf-8"` to `read_text()` |
| Using `metadata: dict = {}` | All Document instances share the same dict — mutations leak between objects | Use `field(default_factory=dict)` |
| Not checking `if text:` for PDFs | `None` or empty strings end up in your content from image-only pages | Always guard with `if text:` before appending |
| Hardcoding file paths | Code only works on your machine | Accept paths as function parameters |
| Skipping `sorted()` in `load_directory` | File order varies across operating systems — tests become flaky | Always `sorted()` the glob results |

## What File Formats Does This Loader Support?

Currently, the document loader handles only **two formats**: `.txt` and `.pdf`. This is intentional — we're keeping it simple for a beginner project.

### How the routing works

Looking at `load_directory()`:

```python
if file_path.suffix == ".pdf":
    doc = load_pdf_file(str(file_path))
else:
    doc = load_text_file(str(file_path))
```

> **Analogy: A Sorting Hat with Only Two Houses**
>
> The loader is like a sorting hat that only knows two categories:
> - "You look like a PDF" → sends you to the PDF handler
> - "You're anything else" → sends you to the plain text handler

### What works, what kind-of works, and what breaks

| File Type | Works? | Why |
|-----------|--------|-----|
| `.txt` | Yes | Designed for this — reads as plain text |
| `.pdf` | Yes | Designed for this — uses `pypdf` to extract text page-by-page |
| `.md`, `.csv`, `.json`, `.html`, `.log` | Partially | These are plain-text-based formats, so `load_text_file()` *can* read them — but it treats them as raw text. It won't understand Markdown headings, CSV columns, HTML tags, or JSON structure. The content loads, but the structure is lost. |
| `.docx` (Word) | No | Binary format — `read_text()` would produce garbled nonsense. Needs `python-docx` library |
| `.xlsx` (Excel) | No | Binary format — needs `openpyxl` library |
| `.pptx` (PowerPoint) | No | Binary format — needs `python-pptx` library |
| Scanned PDFs | No | These are images of text, not actual text. `pypdf` returns empty strings. Needs OCR tools like `pytesseract` |

> **Analogy: The Language Barrier**
>
> Think of file formats like languages. Our mail room worker speaks **English** (plain text) and **French** (PDF). If a letter arrives in **Spanish** (Markdown, CSV) — it's close enough to English that the worker can mostly read it, but misses some nuances. If a letter arrives in **Mandarin** (Word, Excel) — the worker is completely lost and produces gibberish.

### Why only two formats?

This is a **learning project**. Two formats are enough to understand the pattern:

1. Each file type needs its own loading strategy
2. The output is always the same: a `Document` object
3. Adding a new format means writing a new `load_X_file()` function and adding a routing rule

In production RAG systems, you'd either add more loaders yourself or use a framework like **LangChain** or **LlamaIndex** that bundles dozens of pre-built loaders for Word docs, web pages, databases, APIs, and more.

## Where Do the Loaded Documents Get Stored?

This is a very common question — and the answer might surprise you: **nowhere permanently**. The document loader does **not** save anything. It only creates temporary Python objects that live **in memory (RAM)**.

```python
doc = load_text_file("ai.txt")   # 'doc' exists in RAM — temporary!
```

> **Analogy: The Waiter**
>
> The document loader is like a waiter who goes to the kitchen (your hard drive), picks up a plate of food (file contents), places it on a standard tray (Document object), and **hands it to you**. The waiter doesn't store the food anywhere — it's now in *your* hands. If you drop it (the variable goes out of scope or the program ends), it's gone.

### The journey of your data

```
1. ON DISK (permanent)        →  data/sample_texts/ai.txt
                                      │
                                      │  load_text_file() reads it
                                      ▼
2. IN MEMORY (temporary)      →  doc = Document(content="...", metadata={...})
                                      │
                                      │  chunker splits it
                                      ▼
3. CHUNKED IN MEMORY          →  chunks = chunk_by_size(doc)
                                      │
                                      │  vector store persists it
                                      ▼
4. STORED PERMANENTLY         →  vector_store.add(chunks)  ← THIS is where
   (ChromaDB / FAISS)              permanent storage happens
```

The **permanent storage** only happens later at the **vector store** step. The document loader is purely a "read and pass along" module — it reads from disk, wraps the content into a `Document` object in memory, and hands it off. Nothing is saved, nothing is cached.

### But what does "in memory" actually mean?

Your computer has two types of storage:

| | **RAM (Memory)** | **Disk (Storage)** |
|--|---|---|
| **What it is** | Short-term memory | Long-term storage |
| **Speed** | Super fast | Slower |
| **When power is off** | Everything is **lost** | Everything **stays** |
| **Size** | Small (8-32 GB typically) | Large (256 GB - 2 TB) |
| **Examples** | Variables in your running Python program | Files on your hard drive (`.txt`, `.pdf`, photos) |

> **Analogy: Your Desk vs. Your Filing Cabinet**
>
> - **RAM (Memory)** = Your **desk**. When you're working, you pull papers out of the cabinet and spread them on your desk. It's fast to read and scribble on them. But when you leave the office (program ends / computer shuts down), the desk gets cleared. Everything on it is gone.
>
> - **Disk (Storage)** = Your **filing cabinet**. Things stored here survive even when you leave. But it takes time to walk to the cabinet, open the drawer, and find what you need. It's slower.

> **Analogy: Cooking from a Recipe Book**
>
> - The **recipe book** on your shelf = the `.txt` file on disk (permanent)
> - You **open the book and read it** = `load_text_file()` reads the file
> - The recipe is now **in your head** = the `Document` object is in RAM (temporary)
> - You can cook with it, share it, modify your mental version
> - But when you **fall asleep** (program ends), you forget it — the knowledge in your head is gone
> - The **recipe book is still on the shelf** — unchanged, ready to be read again tomorrow

```python
# This variable exists in RAM — fast to access, but temporary
doc = load_text_file("ai.txt")
print(doc.content)  # Works! It's in memory right now

# When this script ends, 'doc' vanishes from RAM
# But "ai.txt" is still safe on your hard drive
```

That's exactly why the **vector store** step exists later in the pipeline — it takes those temporary in-memory objects and **saves them permanently** (back to disk, in a database), so you don't have to re-load and re-process everything each time you want to search.

> It takes files in different formats, opens them, reads them, slaps a standard label (metadata) on each one, and passes them along in a uniform format (`Document`) so that the rest of the pipeline never has to worry about *where* the data came from or *what format* it was in.
>
> Everything downstream just works with `Document` objects — simple, clean, and consistent.
