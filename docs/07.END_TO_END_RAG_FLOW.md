## 1. The Big Picture — A Smoothie-Making Analogy

Think of the entire RAG pipeline like making a smoothie:

| RAG Step | Smoothie Step | Your File |
|----------|--------------|-----------|
| **Document Loading** | Buying whole fruits from the store | `document_loader.py` |
| **Chunking** | Cutting fruits into small pieces | `chunker.py` |
| **Embedding** | Blending pieces into drinkable form | `embedder.py` |
| **Vector Store** | Pouring into labeled cups in the fridge | `vector_store.py` |
| **Retrieval + Generation** | Picking the right cup when someone asks for a flavor | `rag_pipeline.py` |

You can't blend a whole watermelon (loading without chunking), and you can't drink unblended chunks (chunking without embedding). Each step prepares the data for the next.

## 2. Step 1: Document Loading — "Buying the Fruits"

**File**: `rag_beginner_basics/document_loader.py`

This is where raw files on disk become structured Python objects.

### The `Document` dataclass — your "shopping bag"

```python
@dataclass
class Document:
    content: str              # The actual text ("the fruit itself")
    metadata: dict            # Info ABOUT the text ("the label on the fruit")
```

**Analogy**: When you buy an apple, you get the apple (content) AND the receipt/label (metadata: where it's from, how much it weighs, what type).

### `load_text_file()` — reading .txt files

Reads a plain text file and wraps it as a `Document`:

```python
def load_text_file(file_path: str) -> Document:
    with open(file_path, "r", encoding="utf-8") as f:
        content = f.read()

    metadata = {
        "source": Path(file_path).name,        # "machine_learning.txt"
        "file_path": str(Path(file_path)),      # Full path
        "file_type": Path(file_path).suffix,    # ".txt"
        "char_count": len(content),             # 15000 characters
    }
    return Document(content=content, metadata=metadata)
```

**What's happening**: Open file -> read all text -> attach a "name tag" (metadata) -> put in a `Document` bag.

### `load_pdf_file()` — reading PDFs

PDFs are trickier because they're not plain text — they're formatted documents with pages:

```python
def load_pdf_file(file_path: str) -> Document:
    reader = PdfReader(file_path)
    pages = []
    for page in reader.pages:
        text = page.extract_text()    # Pull text from each PDF page
        if text:
            pages.append(text)
    content = "\n\n".join(pages)      # Glue all pages together
```

**Analogy**: A `.txt` file is like a scroll — one continuous text. A PDF is like a book with pages. `load_pdf_file` tears out each page, reads the text, and tapes them together into one long scroll.

### `load_directory()` — bulk loading

```python
def load_directory(directory: str, pattern: str = "*.txt") -> list[Document]:
    files = sorted(Path(directory).glob(pattern))   # Find all matching files
    documents = []
    for file_path in files:
        if file_path.suffix == ".pdf":
            documents.append(load_pdf_file(str(file_path)))
        else:
            documents.append(load_text_file(str(file_path)))
    return documents
```

**Analogy**: Instead of buying one fruit at a time, this is like saying "give me everything in the fruit aisle" — it loads ALL files matching a pattern from a folder.

### Where it's called

In `rag_pipeline.py`, the `ingest()` method kicks it off:

```python
def ingest(self, data_dir: str, pattern: str = "*.txt") -> int:
    documents = load_directory(data_dir, pattern)   # <-- LOADING happens here
    # ... then chunking, embedding, storing
```

## 3. Step 2: Chunking — "Cutting the Fruits"

**File**: `rag_beginner_basics/chunker.py`

### Why chunk at all?

Your sample file `machine_learning.txt` might be 10,000 characters. But:

- Embedding models have token limits
- Smaller chunks = more precise retrieval
- A user asking "what is supervised learning?" doesn't need the ENTIRE document — just the relevant paragraph

**Analogy**: If someone asks "what's in aisle 3?", you don't hand them the entire store map. You tear out just the aisle 3 section. Chunking pre-tears the map into sections so you can quickly hand over the right piece.

### Strategy 1: `chunk_by_size()` — the cookie cutter

```python
def chunk_by_size(document: Document, chunk_size: int = 500, overlap: int = 50):
    text = document.content
    chunks = []
    start = 0
    while start < len(text):
        end = start + chunk_size          # Take exactly 500 chars
        chunk_text = text[start:end]
        chunks.append(Document(content=chunk_text, metadata={...}))
        start += chunk_size - overlap     # Step back 50 chars (overlap)
    return chunks
```

**Analogy — Cutting a baguette**: Imagine cutting bread into 500mm slices. Simple, predictable, but you might cut right through a word or sentence. The `overlap=50` is like making each slice overlap slightly — so no information falls in the crack between two slices.

```
Text:    [aaaaaaaaaa|bbbbbbbbbb|cccccccccc]
                    ^overlap^
Chunk 1: [aaaaaaaaaa|bb]
Chunk 2:          [bb|bbbbbbbbbb|cc]
Chunk 3:                      [cc|cccccccccc]
```

### Strategy 2: `chunk_by_sentences()` — the respectful cutter

```python
def chunk_by_sentences(document: Document, sentences_per_chunk: int = 5, overlap: int = 1):
    sentences = re.split(r'(?<=[.!?])\s+', text)   # Split at sentence boundaries
    for i in range(0, len(sentences), sentences_per_chunk - overlap):
        chunk_sentences = sentences[i:i + sentences_per_chunk]
        chunk_text = " ".join(chunk_sentences)
        chunks.append(Document(content=chunk_text, metadata={...}))
```

**Analogy — Cutting between paragraphs**: Instead of blindly cutting every 500 characters (which might split "machine lear|ning"), this finds sentence endings (periods, question marks) and cuts BETWEEN sentences. Like cutting bread at the natural grooves.

### Strategy 3: `chunk_recursive()` — the smart cutter

```python
separators = ["\n\n", "\n", ". ", " "]   # Try these in order

def chunk_recursive(document, chunk_size=500):
    # Try splitting by paragraph breaks first (\n\n)
    # If chunks are still too big, try line breaks (\n)
    # If still too big, try sentences (. )
    # Last resort: split by spaces
```

**Analogy — Disassembling LEGO**: First try to separate by major sections (paragraphs). If a section is still too big, break it into lines. Still too big? Break into sentences. Last resort: break into words. You always try the biggest natural boundary first.

```
Document: "Chapter 1\n\nParagraph about ML.\nAnother line.\n\nChapter 2\n\n..."
                       ^ Try \n\n first (chapter boundaries)
                              ^ Then \n (line boundaries)
                                           ^ Then ". " (sentence boundaries)
```

### Comparison of the three strategies

| Strategy | Pros | Cons | Best For |
|----------|------|------|----------|
| `chunk_by_size` | Simple, predictable | May split mid-word/sentence | Quick prototyping |
| `chunk_by_sentences` | Respects sentence boundaries | Chunk sizes vary | Narrative text |
| `chunk_recursive` | Preserves semantic structure | More complex logic | Production use |

### Where it's called

In `rag_pipeline.py`:

```python
def ingest(self, data_dir, pattern="*.txt"):
    documents = load_directory(data_dir, pattern)     # Step 1: Load
    all_chunks = []
    for doc in documents:
        chunks = chunk_by_size(doc)                   # <-- CHUNKING happens here
        all_chunks.extend(chunks)
    # ... then embedding, storing
```

Currently the pipeline uses `chunk_by_size` by default. You can swap it for `chunk_by_sentences` or `chunk_recursive` to see how retrieval quality changes.

## 4. Step 3: Embedding — "Blending into Searchable Form"

**File**: `rag_beginner_basics/embedder.py`

### What embedding does

Converts text (human-readable) into a list of numbers (computer-searchable). Two texts with similar meaning produce similar numbers.

```python
class OpenAIEmbedder:
    def __init__(self, model: str = "text-embedding-3-small"):
        self.client = OpenAI()          # Uses OPENAI_API_KEY from .env
        self.model = model              # "text-embedding-3-small"
        self.dimensions = 1536          # Output: 1536 numbers per text
```

### Single text embedding

```python
def embed_text(self, text: str) -> list[float]:
    response = self.client.embeddings.create(
        input=text,
        model=self.model
    )
    return response.data[0].embedding   # -> [0.012, -0.034, 0.089, ... 1536 numbers]
```

**Analogy — DNA test for meaning**: Every chunk gets a unique "DNA fingerprint" of 1536 numbers. Chunks about similar topics have similar DNA. When you ask a question, your question gets its own DNA, and the system finds chunks with the closest DNA match.

```
"Machine learning uses data"  -> [0.82, 0.15, 0.43, -0.21, ... 1536 floats]
"ML trains on datasets"       -> [0.80, 0.16, 0.44, -0.20, ... 1536 floats]  <- very close!
"How to bake a cake"          -> [0.11, 0.67, -0.32, 0.55, ... 1536 floats]  <- very different!
```

### Batch embedding (efficiency)

```python
def embed_batch(self, texts: list[str]) -> list[list[float]]:
    response = self.client.embeddings.create(
        input=texts,           # Send ALL chunks at once
        model=self.model
    )
    return [item.embedding for item in response.data]
```

**Why batch?** If you have 100 chunks, calling the API 100 times is slow (100 network round trips). Batch sends all 100 in one call. Like mailing 100 letters in one package instead of 100 separate trips to the post office.

### Where it's called

In `rag_pipeline.py`:

```python
def ingest(self, data_dir, pattern="*.txt"):
    documents = load_directory(data_dir, pattern)     # Step 1: Load
    for doc in documents:
        chunks = chunk_by_size(doc)                   # Step 2: Chunk
        all_chunks.extend(chunks)

    texts = [chunk.content for chunk in all_chunks]
    embeddings = self.embedder.embed_batch(texts)     # <-- EMBEDDING happens here

    self.vector_store.add(all_chunks, embeddings)     # Step 4: Store
```

And during **query time**, the question also gets embedded with the same model:

```python
def query(self, question: str, top_k: int = 3):
    question_embedding = self.embedder.embed_text(question)  # <-- Same process!
    results = self.vector_store.query(question_embedding, top_k=top_k)
```

**Important**: The question MUST be embedded with the same model as the documents. If you use different models, the "DNA fingerprints" live in different coordinate systems and can't be compared. It's like comparing GPS coordinates from two different maps.

## 5. Step 4: Vector Storage — "Labeled Cups in the Fridge"

**File**: `rag_beginner_basics/vector_store.py`

### The interface — what every vector store must do

```python
class VectorStore(ABC):
    def add(self, documents, embeddings):     # Store chunks + their vectors
    def query(self, query_embedding, top_k):  # Find the N most similar chunks
    def count(self):                          # How many chunks are stored?
```

### ChromaDB — the primary implementation

```python
class ChromaVectorStore(VectorStore):
    def __init__(self, collection_name="rag-basics", persist_dir="./vectordb_data/chromadb"):
        self.client = chromadb.PersistentClient(path=persist_dir)
        self.collection = self.client.get_or_create_collection(
            name=collection_name,
            metadata={"hnsw:space": "cosine"}   # Measure similarity using cosine
        )
```

**What is cosine similarity?** It measures the angle between two vectors:
- Angle = 0 degrees -> identical meaning (score = 1.0)
- Angle = 90 degrees -> completely unrelated (score = 0.0)

**Analogy**: Two arrows pointing the same direction are similar. Two arrows pointing in perpendicular directions are unrelated. Cosine similarity measures how "aligned" the arrows are.

### Adding documents

```python
def add(self, documents, embeddings):
    ids = [hashlib.md5(doc.content.encode()).hexdigest() for doc in documents]
    self.collection.add(
        ids=ids,                # Unique ID (MD5 hash prevents duplicates)
        documents=contents,     # The actual text
        embeddings=embeddings,  # The 1536-dim vectors
        metadatas=metadatas     # Source info, chunk index, etc.
    )
```

**Analogy**: Each cup in the fridge gets a unique barcode (MD5 hash), the smoothie itself (text), a flavor profile (embedding vector), and a label (metadata saying which fruit it came from).

### Querying

```python
def query(self, query_embedding, top_k=3):
    results = self.collection.query(
        query_embeddings=[query_embedding],
        n_results=top_k
    )
    # Returns the top_k chunks whose vectors are closest to the query vector
```

**Analogy**: You walk up to the fridge with a flavor sample (query embedding) and say "find me the 3 cups that taste most like this." The fridge compares your sample against every cup's flavor profile and hands back the best matches.

### FAISS — the alternative implementation

```python
class FAISSVectorStore(VectorStore):
    # Uses Facebook AI Similarity Search
    # Faster for large datasets
    # Stores data in memory + optional disk persistence
```

Same concept, different engine — like choosing between SQLite and PostgreSQL. Both store and retrieve, but with different performance characteristics.

## 6. Step 5: Retrieval and Generation — "Serving the Right Cup"

**File**: `rag_beginner_basics/rag_pipeline.py`

This is where everything comes together. The `query()` method does two things:

### Part A: Retrieval — finding relevant chunks

```python
def query(self, question: str, top_k: int = 3):
    # 1. Convert question to the same vector space as our stored chunks
    question_embedding = self.embedder.embed_text(question)

    # 2. Find the top_k most similar chunks
    results = self.vector_store.query(question_embedding, top_k=top_k)

    # 3. Format the retrieved chunks into a context string
    context_parts = []
    for result in results:
        source = result["metadata"]["source"]
        context_parts.append(f"[Source: {source}]\n{result['content']}")
    context = "\n\n---\n\n".join(context_parts)
```

### Part B: Generation — asking the LLM with context

```python
    # 4. Build the augmented prompt
    prompt = RAG_PROMPT_TEMPLATE.format(context=context, question=question)

    # 5. Call the LLM with the context-enriched prompt
    response = self.llm_client.chat.completions.create(
        model=self.model,              # "gpt-4o-mini"
        messages=[
            {"role": "system", "content": "You are a helpful assistant..."},
            {"role": "user", "content": prompt}
        ],
        temperature=0.2,               # Low randomness for factual answers
        max_tokens=500
    )
```

### The RAG prompt template

```
You are a helpful assistant. Answer based on provided context.
If context lacks information, say so honestly.

## Context
{context}          <-- The retrieved chunks go here

## Question
{question}         <-- The user's original question

## Instructions
- Answer ONLY from context
- Cite source documents
- Be concise
```

**Analogy — An open-book exam**:
- The **system prompt** = exam instructions ("answer only from provided materials")
- The **context** = reference sheets stapled to the exam paper (retrieved chunks)
- The **question** = the actual exam question
- The **LLM response** = the student's answer, grounded in the reference material

Without the context (plain LLM call), the model would rely entirely on its training data — which might be outdated, incomplete, or wrong for your specific documents. With context (RAG), it has the exact relevant passages right in front of it.

## 8. How the Code Connects — Following a Real Request

Let's trace a real example end-to-end through the actual code.

### Ingestion: `python -m rag_beginner_basics ingest --data-dir data/sample_texts`

**Entry point** -> `main.py`:
```python
# 1. Parse CLI args
args = parser.parse_args()

# 2. Create vector store
vector_store = ChromaVectorStore(
    collection_name="rag-basics",
    persist_dir="./vectordb_data/chromadb"
)

# 3. Create pipeline (wires everything together)
pipeline = RAGPipeline(vector_store=vector_store)

# 4. Run ingestion

```

**Inside `pipeline.ingest()`** -> `rag_pipeline.py`:
```python
def ingest(self, data_dir, pattern="*.txt"):
    # LOADING: 4 .txt files -> 4 Document objects
    documents = load_directory(data_dir, pattern)

    # CHUNKING: 4 documents -> ~40 chunks
    all_chunks = []
    for doc in documents:
        chunks = chunk_by_size(doc)
        all_chunks.extend(chunks)

    # EMBEDDING: 40 chunks -> 40 vectors
    texts = [chunk.content for chunk in all_chunks]
    embeddings = self.embedder.embed_batch(texts)

    # STORING: 40 (chunk, vector) pairs -> ChromaDB
    self.vector_store.add(all_chunks, embeddings)

    return len(all_chunks)
```

### Query: `python -m rag_beginner_basics query "What is machine learning?"`

**Entry point** -> `main.py`:
```python
result = pipeline.query(question="What is machine learning?", top_k=3)
print(result["answer"])
```

**Inside `pipeline.query()`** -> `rag_pipeline.py`:
```python
def query(self, question, top_k=3):
    # EMBED the question (same model as ingestion!)
    question_embedding = self.embedder.embed_text(question)

    # RETRIEVE top 3 similar chunks from ChromaDB
    results = self.vector_store.query(question_embedding, top_k=top_k)

    # BUILD context string from retrieved chunks
    context = format_results_as_context(results)

    # GENERATE answer using LLM + context
    prompt = RAG_PROMPT_TEMPLATE.format(context=context, question=question)
    response = self.llm_client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": SYSTEM_PROMPT},
            {"role": "user", "content": prompt}
        ]
    )

    return {
        "question": question,
        "answer": response.choices[0].message.content,
        "sources": [r["metadata"]["source"] for r in results],
        "num_chunks_retrieved": len(results)
    }
```

## 9. Quick Cheat Sheet

### Files and their roles

| What | File | Key Function | Smoothie Analogy |
|------|------|-------------|------------------|
| Load files from disk | `document_loader.py` | `load_directory()` | Buying whole fruits |
| Cut text into pieces | `chunker.py` | `chunk_by_size()` | Slicing fruit into pieces |
| Convert text to numbers | `embedder.py` | `embed_batch()` | Blending into smoothie |
| Store in database | `vector_store.py` | `ChromaVectorStore.add()` | Pouring into labeled cups |
| Search + answer | `rag_pipeline.py` | `query()` | Finding the right cup |
| Wire everything together | `main.py` | CLI entry point | The kitchen counter |

### Key concepts in one sentence each

| Concept | One-Liner |
|---------|-----------|
| **Document** | A Python object holding text content + metadata about that text |
| **Chunk** | A smaller piece of a document, sized for embedding and retrieval |
| **Embedding** | A list of 1536 numbers that represents the "meaning" of a text |
| **Vector Store** | A database that finds documents by meaning-similarity, not keyword match |
| **Cosine Similarity** | Measures how "aligned" two vectors are (1.0 = identical, 0.0 = unrelated) |
| **RAG Prompt** | A prompt that includes retrieved context so the LLM answers from YOUR data |
| **top_k** | How many similar chunks to retrieve (default: 3) |

### The three chunking strategies compared

| Strategy | How It Cuts | Pros | Cons | Best For |
|----------|------------|------|------|----------|
| `chunk_by_size` | Every N characters | Simple, predictable | May split mid-word | Quick prototyping |
| `chunk_by_sentences` | At sentence boundaries | Respects grammar | Chunk sizes vary | Narrative text |
| `chunk_recursive` | Paragraphs -> lines -> sentences | Best semantic preservation | More complex | Production use |

### What happens if you change the knobs?

| Parameter | Smaller Value | Larger Value |
|-----------|--------------|--------------|
| `chunk_size` | More chunks, more precise retrieval, more API calls | Fewer chunks, broader context per chunk, cheaper |
| `overlap` | Risk missing info at boundaries | Safer boundaries, but more redundant data |
| `top_k` | Fewer sources, more focused answer | More sources, broader but potentially noisier |
| `temperature` | More deterministic/factual answers | More creative/varied answers |
